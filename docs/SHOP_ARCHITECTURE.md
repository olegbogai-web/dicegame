# Архитектура магазина (ShopRoom)

## Статус документа

Этот документ задает обязательный архитектурный стандарт для `ShopRoom`: состава товаров, правил генерации, ценообразования и покупок.

Ключевой принцип: `ShopRoom` хранит состояние витрины и ссылки на внешние сервисы, но не содержит в себе прикладную логику генерации и покупки.

---

## 1. Роль магазина в забеге

- Магазин — это отдельная комната (`room_type = shop`).
- В магазин игрок попадает только с глобальной карты.
- При каждом новом посещении магазина витрина генерируется заново.

---

## 2. Обязательная структура витрины

В рамках одного посещения магазин обязан иметь фиксированный набор:

- верхний ряд: `4` карты способностей;
- нижний ряд: `2` артефакта и `2` куба;
- сервисы: `улучшение карты` и `удаление карты`.

Ограничения уникальности в рамках одной витрины:

- способности не повторяются;
- артефакты не повторяются;
- кубы не повторяются.

---

## 3. Правила редкости и генерации

Для способностей, артефактов и кубов:

- редкость определяется случайно по весам;
- после броска редкости выбирается случайный товар этой редкости;
- если валидных товаров нужной редкости нет, разрешен fallback к более низкой редкости;
- итоговый набор обязан удовлетворять правилу «без дублей в своей категории».

Весовые таблицы редкости должны лежать в отдельном policy/config слое, а не в `ShopRoom`.

---

## 4. Экономика и цены

Все товары и сервисы магазина стоят монеты.

Формула цены:

- базовая цена определяется редкостью;
- применяется случайный коэффициент;
- итоговая цена округляется/нормализуется до целого значения.

Обязательные ограничения цены:

- минимум: `1`;
- максимум: `99`;
- цена всегда целое число.

`ShopRoom` не считает цену сам — расчет выполняет внешний `ShopPricingPolicy`.

---

## 5. Правила покупки

- Каждый товар/сервис можно купить только `1` раз за текущее посещение магазина.
- После покупки слот помечается как `sold` и не может быть куплен повторно.
- Покупка возможна только при достаточном количестве монет.
- При успешной покупке монеты списываются сразу.

---

## 6. Сервисы магазина

## 6.1. Улучшение карты

Сервис улучшения в магазине обязан работать по тому же контракту, что и существующий flow награды улучшения карты:

1. выбирается случайная карта игрока, у которой есть доступные улучшения;
2. игрок выбирает один из доступных вариантов улучшения;
3. если карт с доступным новым улучшением нет, выбирается случайная уже улучшенная карта;
4. игроку показываются альтернативные ветки + текущая версия (можно оставить текущую).

`ShopRoom` только инициирует сервис и передает результат в runtime игрока.

## 6.2. Удаление карты

Новая механика: игрок покупает удаление и выбирает `1` свою способность для удаления из runtime-характеристик.

Ограничения:

- удаляется только из текущего runtime-забега;
- удаление не должно оставлять «битые» ссылки в loadout/сохранении;
- сервис удаления также одноразовый за посещение магазина.

---

## 7. Границы ответственности файлов

Ниже — обязательное разделение ответственности.

### Должно быть в `content/rooms/subclasses/shop_room.gd`

- runtime-состояние конкретного магазина (список слотов, sold-флаги, ссылки на цены);
- вызовы внешних сервисов генерации/покупки;
- входные точки lifecycle (`prepare/enter/leave`).

### Не должно быть в `content/rooms/subclasses/shop_room.gd`

- таблиц редкости и контент-пулов;
- формул расчета цены;
- бизнес-логики апгрейда/удаления карт;
- прямого изменения UI-узлов вне presentation-слоя.

### Должно быть в `content/rooms/definitions/shop_room_definition.gd`

- только данные конфигурации магазина (количество слотов, refs policy/пулов, refs visual/layout).

### Не должно быть в `content/rooms/definitions/shop_room_definition.gd`

- рандома, расчетов цены, мутаций runtime-игрока.

### Должно быть в отдельных сервисах (целевая декомпозиция)

- `shop_inventory_generator` — сбор витрины без дублей;
- `shop_rarity_policy` — бросок редкости и fallback;
- `shop_pricing_policy` — расчет и нормализация цены `1..99`;
- `shop_purchase_service` — валидация покупки, списание монет, пометка `sold`;
- `shop_card_upgrade_service` — flow улучшения карты;
- `shop_card_remove_service` — flow удаления карты.

### Не должно быть в UI/presentation

- списания монет;
- валидации доступности покупки;
- изменения runtime-данных игрока напрямую.

---

## 8. Минимальные runtime-контракты

`ShopOffer`:

- `offer_id`;
- `offer_type` (`ability`, `artifact`, `dice`, `card_upgrade_service`, `card_remove_service`);
- `content_ref_id` (для сервисов может быть `null`);
- `rarity`;
- `price`;
- `is_sold`.

`ShopRuntimeState`:

- `room_id`;
- `offers` (ровно 10 слотов: 4 + 2 + 2 + 2 сервиса);
- `generated_at_visit_index`;
- `is_completed`.

## 10. Требования к debug-логам

Для всех будущих изменений магазина обязательны короткие debug-сообщения с префиксом подсистемы.

Минимальный набор точек логирования:

- генерация витрины (что сгенерировано и по какой цене);
- попытка покупки (что выбрано, цена, хватает ли монет);
- успешная покупка/применение сервиса (что выдано/удалено/улучшено);
- причины отклонения/пропуска (пустые опции, отсутствующие данные, отказ по монетам).

Формат логов должен оставаться кратким: одна строка на событие, без длинных дампов UI/ресурсов.

## 11. Текущая реализация `ShopRoomController` (scene-level orchestration)

Для текущей сцены `res://scenes/shop.tscn` добавлен контроллер `res://scenes/scripts/shop_room_controller.gd`.
Он действует как orchestration-слой UI/runtime, не меняя контент-ресурсы способностей/артефактов/кубов.

### Ключевые функции контроллера

- Генерация витрины:
  - `_generate_shop_inventory`;
  - `_generate_ability_offers`;
  - `_generate_artifact_offers`;
  - `_generate_cube_offers`;
  - `_generate_service_offers`.
- Ценообразование:
  - `_compute_ability_price` по формуле `(rarity + 1) * rand(3..8)`;
  - `_compute_artifact_price` по формуле `(rarity + 2) * rand(6..11)`;
  - `_compute_cube_price` по формуле `(rarity + 2) * rand(5..7)`.
- Покупка и валидация:
  - `_try_purchase_offer` (проверка монет, списание, выдача награды, sold-флаг);
  - `_refresh_offers_visual_state` (доступность/недостаток монет/продано).
- Сервисные flow:
  - `_open_upgrade_modal` + `_build_upgrade_modal_entries` + `_resolve_service_selection` для `card_up`;
  - `_open_remove_modal` + `_build_remove_modal_entries` + `_resolve_service_selection` для `card_-`.
- Навигация:
  - `_on_leave_shop_pressed` (выход в сохраненную сцену глобальной карты).

### Переиспользуемые существующие решения

- `PostBattleRewardFlow` используется как provider существующей логики:
  - загрузка контентных пулов (ability/artifact/dice);
  - бросок редкости и fallback к меньшей редкости;
  - helpers по upgrade-веткам способностей.
- `GlobalMapRuntimeState` используется для загрузки/сохранения runtime-игрока и возврата на глобальную карту.
