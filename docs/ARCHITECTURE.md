# Архитектура проекта Dicegame

## Цель архитектуры

Архитектура должна позволять без болезненных переделок расширять игру в следующих направлениях:

- добавление новых состояний;
- добавление новых способностей;
- добавление новых типов кубов;
- добавление новых комнат;
- добавление новых событий;
- добавление новых монстров и боссов;
- добавление новых артефактов;
- добавление новых правил сложности;
- добавление новых мета-улучшений между забегами.

Главный принцип: **данные описывают контент, а код описывает правила исполнения**.

То есть новая способность, новый артефакт или новый монстр должны по возможности добавляться через ресурсы/конфиги и переиспользование уже существующих механик, а не через копирование большой части логики.

## Архитектура подсистемы dice

Документ `docs/DICE_ARCHITECTURE.md` является специализированным архитектурным стандартом для подсистемы кубиков и обязателен для всех изменений в `content/dice/` и связанных интеграциях. Он определяет целевую декомпозицию, правила зависимости модулей и требование, что `content/dice/dice.gd` должен оставаться тонким composition root / facade, а не контейнером прикладной логики.

Зона ответственности этого документа:
- архитектура runtime-подсистемы `dice`;
- разделение ответственности внутри `content/dice/`;
- правила эволюции `dice.gd`;
- требования к будущему рефакторингу dice.

Ссылка: [docs/DICE_ARCHITECTURE.md](./DICE_ARCHITECTURE.md)

## Архитектура подсистемы combat

Документ `docs/COMBAT_ARCHITECTURE.md` является специализированным архитектурным стандартом для боевой подсистемы и обязателен для любых будущих изменений в turn order, боевых кубах, активации способностей, правилах завершения боя и интеграциях с боевыми комнатами. Зона ответственности этого документа:
- архитектура подсистемы `combat`;
- фазовая модель боя и структура хода существа;
- правила расхода боевых кубов;
- контракт активации способностей игрока и монстров;
- границы между `Combat` и `BattleRoom`.

Ссылка: [docs/COMBAT_ARCHITECTURE.md](./COMBAT_ARCHITECTURE.md)

- архитектура подсистемы `monster ai`;
- границы между `Combat` и monster-specific decision logic;
- правила состава AI-профиля отдельного монстра;
- перечень того, что запрещено помещать в AI конкретного монстра.

Ссылка: [docs/MONSTER_AI_ARCHITECTURE.md](./MONSTER_AI_ARCHITECTURE.md)

- границы между доменной логикой артефактов и UI/комнатами/боем.

Документ `docs/CARD_UPGRADE_ARCHITECTURE.md` является специализированным архитектурным стандартом для ветвящихся улучшений способностей игрока и монстров, а также для post-battle flow грани `card_up`. - правила уникальности по `ability_id` для base/upgraded версий;
- границы между reward orchestration, `ability_reward` presentation и combat runtime;
- требования к поддержке upgraded-способностей у монстров.

Ссылка: [docs/CARD_UPGRADE_ARCHITECTURE.md](./CARD_UPGRADE_ARCHITECTURE.md)

## Архитектура событийных комнат (EventRoom)

Документ `docs/EVENT_ROOM_ARCHITECTURE.md` является специализированным архитектурным стандартом для событийных комнат и обязателен для реализации event-сценариев, вариантов выбора и резолва исходов через кубик события. Он фиксирует data-driven модель: у события есть собственные фон/текст, набор вариантов, каждый вариант хранится в отдельном файле и содержит три исхода (позитивный/нейтральный/негативный) с последствиями.

- правила броска event dice по цветным граням выбранного варианта;
- пайплайн применения последствий (артефакты, способности, модификация кубов, запуск боя/следующего события и т.д.);
- требования к валидации событийного контента.

Ссылка: [docs/EVENT_ROOM_ARCHITECTURE.md](./EVENT_ROOM_ARCHITECTURE.md)

## Архитектурные принципы

### 1. Data-driven подход
Весь игровой контент должен быть вынесен в ресурсы/описания:

- `AbilityDefinition`
- `StatusDefinition`
- `ArtifactDefinition`
- `DiceDefinition`
- `MonsterDefinition`
- `RoomDefinition`
- `EventDefinition`
- `DifficultyDefinition`
- `RewardDefinition`

Код должен уметь читать эти определения и исполнять их через общий рантайм.

### 2. Композиция вместо жесткого наследования
Не строить архитектуру вокруг длинных иерархий вроде:

- `FireMonster extends Monster`
- `IceMonster extends Monster`
- `PoisonMonster extends Monster`

Вместо этого:

- у сущности есть базовые характеристики;
- у сущности есть набор компонентов/контейнеров;
- поведение собирается из способностей, состояний, модификаторов, правил ИИ и эффектов.

### 3. Разделение “контент / логика / представление”
Нужно четко разделить:

- **Domain layer** — правила игры;
- **Application layer** — управление потоком забега и экранами;
- **Content layer** — ресурсы с описанием данных;
- **Presentation layer** — UI, сцены, анимации, VFX, звук.

### 4. Событийная модель
Важные игровые действия должны публиковаться как события:

- начало боя;
- бросок куба;
- применение способности;
- получение урона;
- наложение состояния;
- завершение хода;
- смерть сущности;
- вход в комнату;
- получение артефакта.

Это позволит легко подключать:

- артефакты-реакции;
- статусы-реакции;
- пассивные способности;
- аналитику;
- визуальные эффекты.

### 5. Единый пайплайн резолва
Любое действие должно проходить через единый конвейер:

1. Инициатор создает намерение (`ActionIntent`);
2. Система проверяет условия;
3. Применяются модификаторы;
4. Формируется итоговый набор эффектов;
5. Эффекты исполняются последовательно;
6. Публикуются события;
7. Обновляется состояние боя/забега.

Это особенно важно для взаимодействия способностей, артефактов, статусов и сложности.

## Предлагаемые слои проекта

## 1. Domain layer
Слой чистых игровых правил. Здесь не должно быть UI-зависимостей.

### Основные модули домена

#### `run/`
Отвечает за структуру забега:

- состояние текущего забега;
- сид/рандом;
- прогресс по комнатам;
- карта маршрута;
- текущая сложность;
- награды;
- завершение забега;
- начисление мета-очков.

Ключевые сущности:

- `RunState`
- `RunProgress`
- `RunContext`
- `RunRewardCalculator`
- `DifficultyRuntime`

#### `combat/`
Отвечает за бой:

- порядок ходов;
- старт раунда;
- бросок боевых кубов;
- расход кубов;
- использование способностей;
- действия ИИ;
- завершение боя;
- расчет урона, брони, эффектов.

Ключевые сущности:

- `BattleState`
- `Combatant`
- `TurnManager`
- `DicePool`
- `ActionResolver`
- `EffectResolver`
- `BattleEventBus`

#### `rooms/`
Отвечает за типы комнат и переходы:

- обычный бой;
- элитный бой / минибосс;
- событие;
- магазин;
- сундук;
- босс;
- специальные комнаты в будущем.

Ключевые сущности:

- `RoomNode`
- `RoomType`
- `RoomGenerator`
- `RoomResolver`
- `MapDiceResolver`

#### `entities/`
Универсальные сущности игрока и монстров.

Ключевые сущности:

- `EntityId`
- `ActorSnapshot`
- `ActorStats`
- `ActorRuntimeStats`
- `ActorLoadout`
- `ActorTags`

И игрок, и монстр должны использовать одну общую модель сущности, различаясь:

- набором статов;
- набором способностей;
- набором кубов;
- ИИ-профилем;
- флагом стороны (`player` / `enemy`).

#### `abilities/`
Отвечает за активные и пассивные способности.

Ключевые сущности:

- `AbilityDefinition`
- `AbilityInstance`
- `AbilityCost`
- `AbilityCondition`
- `AbilityCooldownRule`
- `AbilityUpgradePath`
- `TargetRule`

#### `statuses/`
Состояния должны быть полноценной подсистемой.

Ключевые сущности:

- `StatusDefinition`
- `StatusInstance`
- `StatusStackRule`
- `StatusDurationRule`
- `StatusTrigger`
- `StatusEffect`

Состояния должны уметь:

- жить N ходов;
- жить до конца боя;
- стакаться;
- обновляться без стака;
- реагировать на события;
- изменять входящий/исходящий урон;
- менять правила использования кубов и способностей.

#### `artifacts/`
Артефакты — это постоянные в рамках забега модификаторы.

Ключевые сущности:

- `ArtifactDefinition`
- `ArtifactInstance`
- `ArtifactTrigger`
- `ArtifactModifier`
- `ArtifactUniquenessRule`

Артефакты должны работать преимущественно через подписку на игровые события и через модификаторы пайплайна.

#### `dice/`
Подсистема кубов должна быть отдельной и универсальной.

Ключевые сущности:

- `DiceDefinition`
- `DiceFaceDefinition`
- `DiceRollResult`
- `DicePoolState`
- `DiceModifier`
- `DiceRequirement`
- `DiceRuleSet`

Нужно разделять:

- **боевые кубы**;
- **кубы карты**.

Это два независимых семейства механик с общим базовым интерфейсом.

#### `ai/`
ИИ монстров не должен быть зашит в самих монстрах.

Ключевые сущности:

- `AiProfile`
- `AiDecisionContext`
- `AiDecision`
- `AiRule`
- `AiPlanner`

ИИ должен принимать решение на основании:

- доступных кубов;
- текущих хп/брони;
- статусов на себе и враге;
- доступных способностей;
- фазы боя;
- поведения архетипа.

#### `meta/`
Мета-прогресс вне забега.

Ключевые сущности:

- `MetaProgressionState`
- `MetaCurrency`
- `MetaUpgradeDefinition`
- `MetaUpgradeEffect`
- `UnlockTable`

## 3. Content layer
Слой игровых данных. Для Godot это удобно хранить в `Resource`-объектах, JSON, `.tres`, `.res` или комбинации.

### Рекомендуемая структура каталога

```text
res://
  content/
    abilities/
    artifacts/
    statuses/
    dice/
    monsters/
    rooms/
    events/
    difficulties/
    rewards/
    meta/
```

Каждый ресурс должен иметь:

- стабильный `id`;
- локализуемое имя;
- описание;
- редкость, если применимо;
- теги;
- конфиг поведения;
- ссылки на визуальные ассеты.

---

## 4. Presentation layer
Слой представления: сцены, UI, анимации, звук, VFX.

### Рекомендуемая структура каталога


## Ключевые сущности и модели

## 1. Сущность бойца

Универсальная модель для игрока и монстра:

```text
Combatant
- id
- side
- base_stats
- runtime_stats
- health
- armor
- dice_pool
- abilities
- statuses
- artifacts/modifiers
- ai_profile? 
- tags
```

### Почему это важно
Единая модель позволяет:

- использовать одинаковую логику урона и статусов;
- давать монстрам те же механики кубов, что и игроку;
- не дублировать боевой код.

---

## 2. Определение способности

```text
AbilityDefinition
- id
- name
- description
- rarity (`обычное`, `необычное`, `редкое`, `уникальное`; базовое значение — `обычное`)
- tags
- cooldown_rule
- usage_limit_rule
- dice_cost
- targeting
- conditions
- effects
- upgrade_links
- visuals
```

### Идея
Способность должна быть набором декларативных условий и эффектов.

Примеры эффектов:

- нанести урон;
- дать броню;
- исцелить;
- наложить статус;
- снять статус;
- добавить/удалить куб;
- перебросить куб;
- преобразовать значение куба;
- клонировать куб;
- изменить комнатный куб;
- добрать временную способность.

### Актуальные контентные способности прототипа (кратко)

- `Усиление` (`ability_id = strengthening`): до `3` кубов, активация доступна при сумме значений выбранных кубов `>= 9`, эффект — наложение состояния `Сила +2` на себя.

---

## 3. Определение статуса

```text
StatusDefinition
- id
- name
- description
- stacking_rule
- duration_rule
- triggers
- modifiers
- visuals
```

Примеры триггеров:

- `on_turn_start`
- `on_turn_end`
- `on_receive_damage`
- `on_deal_damage`
- `on_die_roll`
- `on_ability_used`
- `on_battle_start`
- `on_room_entered`

---

## 4. Определение артефакта

```text
ArtifactDefinition
- id
- name
- description
- rarity
- unique_in_run
- triggers
- passive_modifiers
- visuals
```

### Важное правило
Уникальные артефакты должны проверяться через общий реестр уникальности забега, а не через разрозненные проверки в UI.

---

## 5. Подсистема кубов

### Общая модель

```text
DiceDefinition
- id
- family (battle/map)
- faces
- rarity (`обычное`, `необычное`, `редкое`, `уникальное`; базовое значение — `обычное`)
- tags
- visuals
- intrinsic_effects
```

```text
DiceFaceDefinition
- value
- tags
- embedded_effects
- weight
```

### Что должна уметь подсистема кубов

- поддерживать разные наборы граней;
- поддерживать кубы с неравномерными шансами;
- поддерживать эффекты на гранях;
- поддерживать временные модификаторы;
- поддерживать переброс;
- поддерживать сохранение “заблокированных” кубов;
- поддерживать требования типа “четное значение”, “значение >= 4”, “дубль”, “любые 2 куба”.

Важно: формулировка требований вида “нужен куб со значением 5” всегда означает значение **на верхней грани куба**, определенное соответствующей runtime-функцией получения верхней грани (например, `get_top_face()`).

### Важный интерфейс
Способности не должны зависеть от конкретного типа куба напрямую. Они должны работать через общий интерфейс требований:

- `ExactValueRequirement`
- `RangeRequirement`
- `ParityRequirement`
- `TagRequirement`
- `CountRequirement`
- `PatternRequirement`

Это позволит в будущем вводить нестандартные кубы без переписывания способностей.

Дополнительное ограничение по редкости: сущности редкости `уникальное` могут встретиться в рамках одного забега не более одного раза. Само применение этого ограничения выполняется в подсистеме наград/резолва контента, а не в `AbilityDefinition`/`DiceDefinition`.

## Единый runtime-модификатор

Чтобы не размазывать логику по коду, нужен единый механизм модификаторов.

### Примеры модифицируемых параметров

- исходящий урон;
- входящий урон;
- получаемая броня;
- количество стартовых кубов;
- количество комнатных вариантов;
- шанс выпадения редкости;
- стоимость способности;
- кулдаун способности;
- награда после боя.

### Источники модификаторов

- артефакты;
- состояния;
- сложность;
- временные эффекты комнаты;
- эффекты события;
- мета-улучшения.

### Предлагаемая модель

```text
Modifier
- id
- source_type
- source_id
- priority
- operation (add / mul / override / cap / append)
- target_stat_or_rule
- condition
- duration
```

### Примеры доменных событий

### Зачем это нужно
Такой подход дает возможность легко добавлять:

- комбо-механики;
- журнал боя;
- воспроизведение анимаций по факту завершенного действия;
- запись реплея в будущем.

## Паттерн “интент → валидация → резолв”

Для всех активных действий нужен одинаковый шаблон.

### Пример
Игрок нажимает способность:

1. Создается `UseAbilityIntent`;
2. Проверяется, доступна ли способность;
3. Проверяется кулдаун;
4. Проверяется наличие подходящих кубов;
5. Резервируются выбранные кубы;
6. Собираются модификаторы от артефактов/статусов/сложности;
7. Рассчитываются эффекты;
8. Эффекты применяются в фиксированном порядке;
9. Публикуются события;
10. Обновляется состояние боя.

Такой подход резко упрощает отладку и расширение.

## Расширяемость по типам контента

## 1. Добавление новой способности
Новая способность не должна требовать новой боевой системы.

Должно хватать:

- нового `AbilityDefinition`;
- при необходимости нового типа эффекта или условия;
- регистрации этого эффекта в `EffectResolver`.

## 2. Добавление нового состояния
Должно хватать:

- нового `StatusDefinition`;
- при необходимости нового триггера или модификатора.

## 3. Добавление нового куба
Должно хватать:

- нового `DiceDefinition`;
- описания граней;
- при необходимости новых требований или эффектов граней.

## 4. Добавление новой комнаты
Должно хватать:

- нового `RoomDefinition`;
- нового сценария входа/резолва, если это новый тип поведения;
- отдельной presentation-сцены.

## 5. Добавление нового монстра
Должно хватать:

- нового `MonsterDefinition`;
- списка способностей;
- профиля ИИ;
- набора кубов;
- таблицы наград.

## 6. Добавление нового артефакта
Должно хватать:

- нового `ArtifactDefinition`;
- описания триггеров и модификаторов.

## Редкости и таблицы выпадения

Нужна единая подсистема редкости для:

- способностей;
- артефактов;
- кубов;
- событий.

### Базовые редкости

- `common` — белый;
- `uncommon` — зеленый;
- `rare` — синий;
- `unique` — красный.

### Правила

- `unique` может существовать только 1 раз в рамках забега;
- частоты выпадения должны управляться таблицами, а не if/else в коде;
- сложность, артефакты и события могут модифицировать таблицы выпадения.

## Генерация забега и комнат

### Рекомендуемая модель
Не строить карту как полностью вручную нарисованный граф на раннем этапе.

Базовая версия:

- есть линейно-ветвящийся прогресс;
- после комнаты игрок бросает куб карты;
- система формирует следующий набор допустимых комнат;
- на основе значения грани и правил акта выбирается комната.

### Что нужно учитывать

- защита от плохих последовательностей генерации;
- гарантии появления ключевых комнат;
- контроль частоты магазинов и событий;
- специальные правила перед боссом;
- возможность вводить акты/этажи позже.

### Подсистемы

- `RunGenerator`
- `RoomOfferPolicy`
- `MapDicePolicy`
- `ActProgressionPolicy`
- `BossGateRule`

## Боевая система

### Базовый цикл боя

1. Старт боя;
2. Применение стартовых эффектов;
3. Игрок бросает боевые кубы;
4. Игрок тратит кубы на способности;
5. Когда кубы кончились или игрок завершил ход — ход заканчивается;
6. Срабатывают эффекты конца хода;
7. Монстр выполняет свой ход;
8. Начинается следующий раунд;
9. Повтор до победы или смерти.

### Почему этот цикл удобен
Он естественно масштабируется:

- на дополнительные фазы;
- на нескольких врагов;
- на призыв существ;
- на эффекты “в начале следующего хода”;
- на особые боссы и модификаторы сложности.

## ИИ монстров

ИИ должен быть многоуровневым:

### 1. Тактический уровень
Выбирает наиболее полезную способность сейчас.

### 2. Поведенческий уровень
Определяет стиль монстра:

- агрессивный;
- защитный;
- контроль;
- накопительный;
- берсерк;
- боссовый сценарный.

### 3. Сценарный уровень
Для боссов и уникальных врагов:

- фазы боя;
- scripted patterns;
- реакции на порог HP;
- специальные события арены.

Необходимо заложить возможность сочетания rule-based ИИ и scripted-поведения.

---

## Сохранения

Сразу стоит разделить два вида сохранения:

### 1. Meta save
Хранит:

- валюту;
- открытые улучшения;
- открытый контент;
- статистику;
- настройки.

### 2. Run save
Хранит:

- текущий забег;
- текущую комнату;
- текущее состояние боя;
- сид;
- полученные артефакты;
- набор способностей;
- модифицированные кубы;
- уникальности, уже занятые в забеге.

---

## Что важно не делать

### 1. Не хранить игровую логику внутри UI-сцен
UI должен только отображать и отправлять команды.

### 2. Не зашивать контент в код
Новые способности/монстры/артефакты не должны требовать изменения десятков файлов.

### 3. Не смешивать map dice и battle dice в одну неявную систему
У них общая база, но разный контекст, правила и UX.

### 4. Не разбрасывать формулы по проекту
Формулы урона, брони, редкости и наград должны жить в централизованных сервисах/резолверах.

## Рекомендуемый порядок реализации

### Этап 1. Каркас домена
Сначала реализовать:

- базовые сущности игрока и монстра;
- боевой цикл;
- боевые кубы;
- простые способности;
- статусы;
- базовый `EffectResolver`;
- простой бой 1 на 1.

### Этап 2. Забег
Затем добавить:

- `RunState`;
- выбор сложности;
- куб карты;
- базовые типы комнат;
- награды;
- завершение забега.

### Этап 3. Контентные системы
Далее добавить:

- артефакты;
- магазин;
- события;
- редкости;
- улучшения способностей;
- разные типы кубов.

### Этап 4. Метапрогресс
После этого:

- главное меню;
- валюта;
- дерево улучшений;
- разблокировки;
- сейвы.

### Этап 5. Гибкость и масштабирование
В конце расширять:

- боссовые фазы;
- элитные модификаторы;
- несколько врагов;
- продвинутый ИИ;
- дополнительные акты.

## Итог

Базовая архитектура должна строиться вокруг следующих опор:

- единый доменный runtime;
- data-driven контент;
- единая система эффектов и модификаторов;
- событийная модель;
- разделение run/combat/meta;
- единые модели игрока и монстров;
- независимые, но совместимые подсистемы боевых и комнатных кубов.

Если держаться этой структуры, проект можно будет спокойно расширять новыми механиками без постоянного переписывания базового кода.
