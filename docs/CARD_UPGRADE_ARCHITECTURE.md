# Архитектура улучшения карт (Card Upgrade)

## Статус документа

Этот документ задает архитектурный стандарт для ветвящихся улучшений способностей игрока и монстров.

Цель: добавить `card_up` flow без дублирования существующих систем наград и боя, максимально переиспользуя текущий `ability_reward` pipeline.

Связанные документы:
- [docs/REWARD_AND_MONEY_DICE_ARCHITECTURE.md](./REWARD_AND_MONEY_DICE_ARCHITECTURE.md)
- [docs/COMBAT_ARCHITECTURE.md](./COMBAT_ARCHITECTURE.md)
- [docs/SHOP_ARCHITECTURE.md](./SHOP_ARCHITECTURE.md)

---

## 1. Инварианты системы улучшений

1. У способности может быть `0..3` вариантов следующего улучшения.
2. Улучшение меняет только параметры способности (например, `damage`, dice-условия, стоимость), но не базовую роль способности.
3. У улучшенной версии могут быть свои дальнейшие улучшения (цепочка/дерево).
4. Спрайт (asset) у улучшенных версий не меняется.
5. Отображаемое имя строится от **глубины в дереве апгрейдов**:
   - base (`upgrade_level = 0`): `Название`;
   - апгрейд, который выбирается **напрямую из base**: `Название+`;
   - апгрейд, который выбирается из уже улучшенной версии (`+`): `Название++`;
   - и далее по той же логике.

   Важно: если у base есть несколько **параллельных** вариантов апгрейда, все эти варианты имеют один `+` в имени (а не `++`).
6. Все версии одной и той же способности (base + upgrade-ветки) имеют один и тот же `ability_id`.
7. Благодаря общему `ability_id` игрок не может получить базовую способность второй раз как «новую карту».
8. Улучшенные способности хранятся в runtime-статах игрока там же, где обычные способности (без отдельного контейнера «только для апгрейдов»).
9. Улучшенные способности обязаны без специальных кейсов работать в бою через текущий runtime резолв способностей.
10. Монстры также могут использовать улучшенные версии способностей по тому же контракту.

---


### 1.1. Политика именования для ветвлений

- Количество `+` в `display_name` зависит **только от глубины**, а не от количества вариантов в ветке.
- Если base имеет 2-3 альтернативы, и все они выбираются сразу из base, они именуются как `Название+`.
- Суффикс `++` допустим только для версий, которые выбираются уже после выбора версии `Название+`.
- Для параллельных апгрейдов одного уровня различия отражаются в `description`, `effects`, `cost`, `conditions`, но не в количестве `+`.

## 2. Модель данных (data-driven)

### 2.0. Актуальные ветки улучшений (кратко)

- `common_attack`:
  - `1` → `common_attack_upgrade_power`;
    - `1.1` → `common_attack_upgrade_power_damage_12`;
    - `1.2` → `common_attack_upgrade_power_strength_1`.
  - `2` → `common_attack_upgrade_any_die`;
    - `2.1` → `common_attack_upgrade_any_die_random_reroll`;
    - `2.2` → `common_attack_upgrade_any_die_damage_8`.
- `heal`:
  - `1` → `heal_upgrade_power`;
    - `1.1` → `heal_upgrade_power_regeneration_3` (без прямого хила, `Регенерация +3`);
    - `1.2` → `heal_upgrade_power_strength_1_heal_5` (`хил 5` + `Сила +1`);
    - `1.3` → `heal_upgrade_power_heal_12` (`хил 12`).
  - `2` → `heal_upgrade_efficiency`;
    - `2.1` → `heal_upgrade_efficiency_heal_5` (`хил 5`);
    - `2.2` → `heal_upgrade_efficiency_holiness_1_heal_2` (`хил 2` + `Святость +1`).
- `death_blow`:
  - `1` → `death_blow_upgrade_1` (`урон 20`, 1 куб `6`);
    - `1.1` → `death_blow_upgrade_1_1` (`урон 25`);
    - `1.2` → `death_blow_upgrade_1_2` (`урон 15`, при убийстве цели `хил 15`);
    - `1.3` → `death_blow_upgrade_1_3` (`урон 20`, 1 куб `5` или `6`).
  - `2` → `death_blow_upgrade_2` (`урон 30`, 2 куба `6`);
    - `2.1` → `death_blow_upgrade_2_1` (`урон 75`, ровно 3 куба `6`);
    - `2.2` → `death_blow_upgrade_2_2` (`урон 50` + `Обезоружен 1`);
    - `2.3` → `death_blow_upgrade_2_3` (`урон 60`).
- `clawed_series`:
  - base (`clawed_series`) — `3` куба со значением `<= 3`, `5` ударов по `3`.
  - `1` → `clawed_series_upgrade_1` (`5` ударов по `4`);
    - `1.1` → `clawed_series_upgrade_1_1` (`7` ударов по `4`);
    - `1.2` → `clawed_series_upgrade_1_2` (`5` ударов по `6`).
  - `2` → `clawed_series_upgrade_2` (`3` удара по `10`);
    - `2.1` → `clawed_series_upgrade_2_1` (`3` удара по `10` + `Кровотечение 5`);
    - `2.2` → `clawed_series_upgrade_2_2` (`3` удара по `15`).
- `needle_throw` / «Выстрел из мушкета»:
  - base (`needle_throw`) — `1` любой куб, урон `floor(1.5 * значение_куба)`.
  - `1` → `needle_throw_upgrade_1` (урон `floor(2.25 * значение_куба)`);
    - `1.1` → `needle_throw_upgrade_1_1` (урон `3 * значение_куба`);
    - `1.2` → `needle_throw_upgrade_1_2` (урон `5 * значение_куба`, `cooldown_turns = 1`, фактически 1 раз за ход).
  - `2` → `needle_throw_upgrade_2` (принимает только кубы `<= 3`, урон `4 * значение_куба`);
    - `2.1` → `needle_throw_upgrade_2_1` (принимает только кубы `1..2`, урон `8 * значение_куба`);
    - `2.2` → `needle_throw_upgrade_2_2` (принимает только кубы `<= 3`, урон `5 * значение_куба`).
- `crushing_shot` / «Сокрушающий выстрел»:
  - base (`crushing_shot`) — ровно `2` куба с суммой `7`, урон `15`.
  - `1` → `crushing_shot_upgrade_1` (урон `20`);
    - `1.1` → `crushing_shot_upgrade_1_1` (урон `25`);
    - `1.2` → `crushing_shot_upgrade_1_2` (урон `15`, затем `Уязвимость 1` цели).
  - `2` → `crushing_shot_upgrade_2` (ровно `2` куба с суммой `>= 7`, урон `15`);
    - `2.1` → `crushing_shot_upgrade_2_1` (ровно `2` куба с суммой `>= 5`, урон `15`);
    - `2.2` → `crushing_shot_upgrade_2_2` (урон `15 + сумма выбранных кубов`).
- `strengthening` / «Усиление»:
  - base (`strengthening`) — до `3` кубов, сумма `>= 9`, `Сила +3`.
  - `1` → `strengthening_upgrade_1` (сумма `>= 7`, `Сила +3`);
    - `1.1` → `strengthening_upgrade_1_1` (сумма `>= 6`, `Сила +3`);
    - `1.2` → `strengthening_upgrade_1_2` (сумма `>= 7`, `Сила +4`).
  - `2` → `strengthening_upgrade_2` (сумма ровно `9`, `Сила +5`);
    - `2.1` → `strengthening_upgrade_2_1` (сумма ровно `9`, `Сила +6`);
    - `2.2` → `strengthening_upgrade_2_2` (сумма ровно `12`, `Сила +9`).
  - `3` → `strengthening_upgrade_3` (сумма `>= 9`, `Сила +4`);
    - `3.1` → `strengthening_upgrade_3_1` (сумма `>= 9`, `Сила +5`);
    - `3.2` → `strengthening_upgrade_3_2` (сумма `>= 9`, максимум `2` куба, `Сила +6`).
- `reroll_all` / «Переброс всего»:
  - base (`reroll_all`) — переброс всех оставшихся кубов, `once_per_battle`;
  - `1` → `reroll_all_upgrade_1` (`2` раза за бой, не чаще `1` раза за ход);
    - `1.1` → `reroll_all_upgrade_1_1` (`3` раза за бой, не чаще `1` раза за ход);
    - `1.2` → `reroll_all_upgrade_1_2` (`2` раза за бой без лимита на ход).
  - `2` → `reroll_all_upgrade_2` (добавляет 1 случайную копию куба с доски);
    - `2.1` → `reroll_all_upgrade_2_1` (добавляет 2 случайные копии);
    - `2.2` → `reroll_all_upgrade_2_2` (не перебрасывает кубы `>= 5`).
- `reroll` / «Переброс кубика»:
  - base (`reroll`) — 1 раз за ход, перебрасывает выбранный куб;
  - `1` → `reroll_upgrade_1` (до `2` использований за ход);
    - `1.1` → `reroll_upgrade_1_1` (до `3` использований за ход);
    - `1.2` → `reroll_upgrade_1_2` (до `2` использований за ход + лечит на значение переброшенного куба).
  - `2` → `reroll_upgrade_2` (без отката, но требует `1` любой куб на активацию);
    - `2.1` → `reroll_upgrade_2_1` (как `2` + накладывает `Регенерация 2` на владельца);
    - `2.2` → `reroll_upgrade_2_2` (как `2` + снимает `1` стак случайного отрицательного состояния с владельца).
- `poison_injection` / «Отравляющий укол»:
  - base (`poison_injection`) — `1` куб со значением `1`, накладывает `Яд 2` на цель.
  - `1` → `poison_injection_upgrade_1` (`Яд 3`);
    - `1.1` → `poison_injection_upgrade_1_1` (`Яд 4`);
    - `1.2` → `poison_injection_upgrade_1_2` (требует ровно `2` куба `1`, накладывает `Яд 4` на цель и `Отравитель 1` на владельца).
  - `2` → `poison_injection_upgrade_2` (стоимость `1` или `2`, накладывает `Яд 2`);
    - `2.1` → `poison_injection_upgrade_2_1` (стоимость `1..3`, накладывает `Яд 2`);
    - `2.2` → `poison_injection_upgrade_2_2` (стоимость: любой `1` куб, накладывает `Яд 1`).
- `protection_spell` / «Заклинание защиты»:
  - base (`protection_spell`) — `1` куб номиналом `3` или `5`, накладывает `Броня +4` на владельца.
  - `1` → `protection_spell_upgrade_1` (`Броня +6`);
    - `1.1` → `protection_spell_upgrade_1_1` (`Броня +9`);
    - `1.2` → `protection_spell_upgrade_1_2` (`Броня +3` + `Стойкость 1` владельцу).
  - `2` → `protection_spell_upgrade_2` (стоимость: `1` нечетный куб, `Броня +4`);
    - `2.1` → `protection_spell_upgrade_2_1` (стоимость: `1` любой куб, `Броня +4`);
    - `2.2` → `protection_spell_upgrade_2_2` (стоимость: `1` нечетный куб, `Броня +6`).

## 2.1. Что должно быть в ability content

В контенте способности (definition/resource) должен быть отдельный блок метаданных улучшения:

- `ability_id` — стабильный общий id семейства способности;
- `upgrade_level` — уровень текущей версии (`0` для base);
- `follow_up_ability_ids` — список ссылок на доступные следующие версии (0..3);
- параметры самой версии (урон, стоимость, условия и т.д.).

`follow_up_ability_ids` указывает только на **следующий шаг** из текущей ноды дерева, чтобы поддерживать ветвление.

## 2.2. Что не должно быть в ability content

- UI-ссылок на конкретные scene/control;
- императивной логики выбора апгрейда;
- логики выдачи наград после боя;
- отдельного «нового id» для upgraded версии.

---

## 3. Runtime-представление у игрока и монстров

## 3.1. Должно быть

- В runtime loadout хранится текущая выбранная версия способности (с текущими параметрами) и ее `upgrade_level`.
- Контейнер способностей оперирует семейным `ability_id` как ключом уникальности.
- Замена base -> upgraded выполняется как обновление текущей версии способности в слоте/контейнере, а не как добавление «еще одной новой карты».

## 3.2. Не должно быть

- параллельного хранения base и upgraded как двух разных способностей одного семейства;
- отдельного runtime-контура «улучшенные способности», дублирующего основной контейнер;
- боевых проверок вида `if ability_name == ...+`.

---

## 4. Поток награды `card_up` (переиспользование ability_reward)

## 4.1. Источник триггера

- Когда на reward-кубе выпадает `card_up` (`REWARD_CARD_UP_ICON`), запускается flow улучшения способности.

## 4.2. Алгоритм выбора

1. Из текущих способностей игрока формируется пул карт, у которых есть минимум один доступный `follow_up_ability_ids`.
2. Случайно выбирается 1 способность из этого пула.
3. Игроку показываются **все** доступные варианты улучшения выбранной способности (1..3).
4. Игрок выбирает один вариант.
5. В runtime-статах игрока текущая версия способности заменяется на выбранный upgrade-вариант (тот же `ability_id`, новый `upgrade_level`, новые параметры).

Короткая актуализация текущего поведения:
- если у игрока нет способностей с прямыми `follow_up_ability_ids` (например, все карты уже в upgraded-версиях), flow `card_up` не отменяется;
- в таком случае выбирается случайная способность из текущего loadout и показываются параллельные варианты апгрейда её семейства (ветки от base-версии);
- в список опций включается и текущая выбранная версия, чтобы игрок мог оставить текущий вариант или переключиться на другой.

## 4.3. UI/Presentation правило

Визуально `card_up` должен выглядеть так же, как выдача новой способности по `card_+`:

- используется тот же экран/сцена `ability_reward`;
- используются те же базовые функции рендера карточек и выбора игрока;
- различается только источник списка опций (для `card_+` — генерация новых карт, для `card_up` — upgrade-ветка конкретной карты).
- рамка карточки способности (`base_frame`) выбирается строго по редкости:
  - `обычное` или неизвестная редкость -> `base_frame_normal`;
  - `необычное` -> `base_frame_copper_`;
  - `редкое` -> `base_frame_silver`;
  - `уникальное` -> `base_frame_gold`.

Это обязательное требование для минимизации новой логики и рисков расхождения UI-потоков.

## 5. Границы ответственности файлов

## Должно быть

### В `content/abilities/resources/*`, `content/abilities/definitions/*`
- Только данные версии способности и ее upgrade-связей.
- Параметры эффектов/стоимостей/условий конкретной версии.

### В reward/orchestration слое (`post_battle_reward_flow`, battle reward orchestration)
- Триггер `card_up`;
- выбор случайной улучшаемой способности;
- подготовка списка upgrade-опций для `ability_reward`.
- единая функция `_resolve_base_frame_by_rarity` для расчета `base_frame` ассета в `card_up`/`card_+` карточках.

### В `ability_reward` presentation
- Переиспользуемый рендер списка вариантов;
- единый обработчик выбора варианта, который получает уже подготовленные опции;
- применение рамки через `_apply_reward_card_base_frame_visual` без изменения остальной логики карточки.

### В combat runtime
- Исполнение выбранной версии способности без знания, base это или upgraded;
- единый pipeline для игрока и монстров.

## Не должно быть

### В `ability_reward` UI
- Логики случайного выбора карты для апгрейда;
- правил фильтрации upgrade-кандидатов;
- доменной логики мутации player runtime.

### В ability runtime/combat
- Ветвлений по суффиксу `+` в названии;
- специальных «режимов» для upgraded-версий;
- дублированных executor-ов только для апгрейдов.

### В reward flow
- Создания отдельной новой UI-сцены только для `card_up`, если достаточно существующего `ability_reward`.

---

## 6. Контракты совместимости

1. `card_+` (новая способность) продолжает проверять уникальность по `ability_id`.
2. `card_up` не создает новый `ability_id`, а обновляет текущую версию внутри того же семейства.
3. Сохранения/рантайм снимки должны сериализовать текущую версию способности (или upgrade_level+ссылку на definition) так же, как и обычные способности.
4. MonsterDefinition может ссылаться на upgraded-версии способностей напрямую, без отдельного monster-only механизма.

---

## 7. Что этот документ покрывает и что не покрывает

Покрывает:
- архитектуру ветвящихся улучшений способностей;
- границы ответственности content/runtime/reward/UI;
- обязательное переиспользование `ability_reward` flow для `card_up`.

Не покрывает:
- точные балансные числа (урон, веса, вероятности);
- конкретный UX-текст кнопок и локализацию;
- визуальные эффекты и анимации.
