# Архитектура подсистемы Room

## Статус документа

Этот документ является **обязательным архитектурным стандартом** для будущей реализации `Room` и всех ее специализированных подтипов.

Любая реализация комнат **должна следовать** этому документу. Если в коде появляется решение, противоречащее правилам ниже, такое решение считается архитектурным отклонением и должно быть отдельно обосновано.

Документ описывает:

- целевую модель базового класса `Room`;
- допустимую декомпозицию данных, правил, состояний и визуала комнаты;
- правила расширения для новых типов комнат;
- требования к подклассам `ShopRoom`, `EventRoom`, `BattleRoom`;
- принцип, по которому комнаты **не хранят исполняемую игровую логику**, а содержат только ссылки на обработчики, правила и сценарии.

---

## 1. Цель подсистемы Room

Подсистема `Room` отвечает за представление комнаты как **узла игрового забега**, который:

- знает свой тип;
- знает, какие объекты и сущности присутствуют в комнате;
- знает, какие правила и ограничения применяются внутри комнаты;
- знает текущее runtime-состояние комнаты;
- инициирует запуск и завершение жизненного цикла комнаты;
- связывает доменную комнату с визуальным представлением;
- передает исполнение логики внешним системам через ссылки на обработчики и резолверы.

Комната является **оркестратором данных комнаты**, но не контейнером бизнес-логики.

Ключевой принцип: `Room` хранит **описание**, **runtime-состояние** и **ссылки на логику**, но не реализует прикладные сценарии внутри себя.

---

## 2. Почему это важно

Это решение нужно, чтобы поддержать цели из `GAME_VISION` и общей `ARCHITECTURE`:

- маршрут должен легко расширяться новыми типами комнат;
- события должны менять состояние забега, а не быть одноразовыми hardcode-сценами;
- магазин должен быть частью общей экономики забега;
- боевые комнаты должны интегрироваться с боевой подсистемой, но не дублировать ее;
- прикладная логика боевой комнаты должна быть описана отдельно и опираться на единый контракт боя;
- архитектура monster AI, включая ожидание остановки кубов и общий execution pipeline способностей, должна быть вынесена из `BattleRoom` и `battle_table` в отдельные сервисы;
- визуал комнаты должен быть заменяемым без переписывания доменной логики;
- новые комнаты должны добавляться через новые определения и новые обработчики, а не через разрастание одного большого класса.

---

## 3. Архитектурные принципы подсистемы Room

### 3.1. Room — это доменная точка входа в комнату

`Room` представляет конкретную комнату текущего забега.

Она должна отвечать за:

- идентичность комнаты;
- тип комнаты;
- список объектов комнаты;
- активные правила комнаты;
- runtime-состояние комнаты;
- ссылки на жизненный цикл комнаты;
- ссылки на визуальные ресурсы и визуальные контроллеры;
- связь комнаты с контекстом забега.

`Room` **не должна**:

- содержать полный код генерации контента;
- содержать код боевой системы;
- содержать код UI-экранов;
- хранить в себе функции с бизнес-логикой магазина, события или боя;
- напрямую реализовывать длинные ветки `if/else` по всем типам комнат.

### 3.2. Логика комнаты живет вне Room

Вся прикладная логика должна быть вынесена в отдельные сервисы/обработчики.

Примеры допустимых внешних зависимостей:

- `RoomRuleResolver`
- `RoomLifecycleHandler`
- `RoomStateTransitionPolicy`
- `RoomObjectSpawner`
- `RoomVisualBinder`
- `ShopInventoryGenerator`
- `ShopPricingPolicy`
- `EventScenarioResolver`
- `EventChoiceResolver`
- `BattleEncounterResolver`
- `BattleRoomPresentationBinder`

`Room` и ее подклассы могут хранить:

- идентификаторы обработчиков;
- ссылки на definitions/resources;
- ссылки на policy objects;
- ссылки на scene/template/prefab;
- ссылки на scripted action references;
- ссылки на event hooks.

Но они **не должны хранить** сами исполняемые функции как встроенную прикладную реализацию комнаты.

### 3.3. Data-driven first

Комнаты должны расширяться прежде всего через данные:

- `RoomDefinition`
- `ShopRoomDefinition`
- `EventRoomDefinition`
- `BattleRoomDefinition`
- `RoomRuleDefinition`
- `RoomVisualDefinition`
- `RoomObjectDefinition`
- `RoomLifecycleDefinition`

Подклассы нужны для ясности модели и типобезопасного runtime-контракта, но основной контент должен приходить из definitions.

### 3.4. Разделение на 4 слоя

Подсистема Room должна быть разделена на четыре зоны:

1. **Definition layer** — статические описания комнаты.
2. **Runtime layer** — состояние конкретной комнаты в конкретном забеге.
3. **Resolver/Handler layer** — исполнение логики по ссылкам из комнаты.
4. **Presentation layer** — сцены, виджеты, VFX, текстовые панели, визуальный layout.

`Room` находится на границе между Definition layer и Runtime layer, но вызывает логику только через Handler layer.

## 4. Базовая модель Room

## 4.1. Рекомендуемая роль класса

Базовый класс `Room` должен быть абстракцией верхнего уровня для любой комнаты забега.

Его главная задача — предоставить единый контракт для систем:

- генерации маршрута;
- входа в комнату;
- показа комнаты;
- применения правил комнаты;
- завершения комнаты;
- выдачи наград;
- перехода к следующему шагу забега.

## 4.2. Обязательные зоны ответственности Room

`Room` должна агрегировать следующие данные:

### A. Идентичность
- `room_id`
- `room_type`
- `room_definition_id`
- `run_node_id` / ссылка на узел маршрута

### B. Список объектов комнаты
Список доменных объектов, которые присутствуют в комнате.

Примеры:
- враги;
- торговые слоты;
- интерактивные точки;
- декорации, влияющие на правила;
- контейнеры наград;
- временные room modifiers.

Важно: комната хранит **runtime-представления объектов** или ссылки на их definitions/instances, но не логику этих объектов.

### C. Тип комнаты
Минимум:
- `battle`
- `event`
- `shop`

С расчетом на расширение:
- `elite_battle`
- `boss_battle`
- `treasure`
- `rest`
- `special`
- `scripted`

### D. Правила комнаты
Набор ссылок на правила, которые должны быть применены к комнате.

Примеры:
- запрет на определенные действия;
- модификаторы стоимости;
- ограничения на награды;
- особые правила входа;
- временные эффекты комнаты;
- условия завершения комнаты.

### E. Состояние комнаты
Минимальный жизненный цикл:
- `created`
- `prepared`
- `entered`
- `active`
- `resolving`
- `completed`
- `failed`
- `abandoned`

### F. Запуск и завершение событий комнаты
Комната должна содержать ссылки на точки жизненного цикла:
- `on_prepare_ref`
- `on_enter_ref`
- `on_activate_ref`
- `on_complete_ref`
- `on_leave_ref`

Это не сами функции, а идентификаторы/ссылки, которые исполняются через внешний `RoomLifecycleHandler`.

### G. Визуал комнаты
Комната должна содержать ссылки на визуальные ресурсы:
- `visual_definition_id`
- `scene_ref`
- `layout_ref`
- `theme_ref`
- `ambient_audio_ref`
- `vfx_profile_ref`

Визуал не должен влиять напрямую на доменные решения комнаты, кроме случаев, когда визуальная конфигурация явно отражает доменное состояние.

## 5. Рекомендуемая декомпозиция сущностей

Чтобы `Room` не разрасталась, ее модель должна быть собрана из нескольких малых сущностей.

### 5.1. RoomDefinition
Статическое описание типа и базовой конфигурации комнаты.

Содержит:
- базовый `room_type`;
- список поддерживаемых rule refs;
- lifecycle refs;
- visual refs;
- базовые object slots;
- таблицы генерации или ссылки на них.

### 5.2. RoomState
Runtime-состояние конкретной комнаты в текущем забеге.

Содержит:
- текущую фазу жизненного цикла;
- флаги завершения;
- временные модификаторы;
- факт посещения;
- runtime-переменные, относящиеся только к комнате.

### 5.3. RoomRuleSet
Контейнер ссылок на активные правила комнаты.

Содержит:
- список `rule_ref`;
- порядок применения правил;
- приоритеты;
- область действия правила (`entry`, `active`, `reward`, `exit`).

### 5.4. RoomObjectSet
Контейнер объектов комнаты.

Содержит:
- список `RoomObjectInstance`;
- object slots;
- размещение объектов;
- состояние доступности объекта.

### 5.5. RoomVisualConfig
Описание визуального представления комнаты.

Содержит:
- ссылку на сцену;
- layout-токены;
- фон;
- освещение/тему;
- UI anchors;
- binding refs для презентационного слоя.

### 5.6. RoomLifecycleRefs
Контракт ссылок на жизненный цикл комнаты.

Содержит:
- `prepare_ref`
- `enter_ref`
- `activate_ref`
- `complete_ref`
- `leave_ref`
- при необходимости `fail_ref`

## 6. Обязательный контракт: только ссылки на логику

Это ключевое правило для всей подсистемы.

### 6.1. Что разрешено хранить в Room и подклассах

Разрешено хранить:

- идентификаторы правил;
- идентификаторы сценариев;
- идентификаторы таблиц генерации;
- ссылки на definitions/resources;
- ссылки на визуальные префабы/сцены;
- ссылки на lifecycle handlers;
- runtime-данные, полученные в результате работы внешних систем.

### 6.2. Что запрещено хранить в Room и подклассах

Запрещено хранить:

- код генерации ассортимента магазина;
- код расчета цены;
- код выбора исхода события;
- код резолва выбора игрока;
- код создания вражеской пачки;
- код запуска боя;
- код UI-отрисовки и анимации;
- анонимные callback-функции как основную архитектурную форму логики комнаты.

### 6.3. Как должен выглядеть вызов логики

Правильная схема:

1. `Room` содержит `*_ref` или `*_definition_id`.
2. Внешний orchestrator или resolver получает эти ссылки.
3. Находит соответствующий handler/policy/service.
4. Исполняет логику в отдельном модуле.
5. Возвращает результат в `RoomState`, `RunState` или другой runtime-контейнер.

Неправильная схема:

- `Room` сама генерирует магазин;
- `EventRoom` сама вычисляет исход варианта;
- `BattleRoom` сама реализует боевой цикл;
- комната хранит внутри себя функции-обработчики как основной способ моделирования поведения.

---

## 7. Подклассы Room

Подклассы нужны для специализации структуры данных и ссылок, а не для переноса всей логики в наследование.

Каждый подкласс:

- расширяет контракт данных;
- вводит собственные typed refs;
- задает обязательные definition refs;
- остается тонким контейнером данных и ссылок.

## 8. ShopRoom

## 8.1. Назначение

`ShopRoom` описывает комнату магазина, где игрок получает доступ к экономическому взаимодействию забега.

Она отвечает за:

- ссылки на генерацию ассортимента артефактов и других товаров;
- правила магазина;
- места размещения торговых объектов;
- цены и модификаторы цены через ссылки на policy/resolver;
- визуальное представление магазина.

## 8.2. Что должно находиться в ShopRoom

### Обязательные данные
- `shop_pool_ref` — ссылка на таблицу/источник генерации товаров;
- `inventory_generation_ref` — ссылка на генератор ассортимента;
- `pricing_policy_ref` — ссылка на политику цены;
- `shop_rule_refs` — ссылки на правила магазина;
- `shop_slot_layout_ref` — ссылка на схему размещения товаров;
- `currency_rule_ref` — ссылка на правило валюты/оплаты;
- `shop_visual_ref` — ссылка на визуальную конфигурацию магазина.

### Runtime-данные
- список сгенерированных `ShopOfferInstance`;
- состояние выкупленных/заблокированных слотов;
- временные модификаторы цен;
- флаг обновления ассортимента.

## 8.3. Что не должно находиться в ShopRoom

Не должно быть:

- функции `generate_artifacts()` внутри комнаты;
- функции `calculate_price()` внутри комнаты;
- функции применения покупки внутри комнаты;
- знания о полном UI магазина.

## 8.4. Внешние системы для ShopRoom

Рекомендуемые внешние сервисы:

- `ShopInventoryGenerator`
- `ShopOfferResolver`
- `ShopPricingPolicy`
- `ShopPurchaseHandler`
- `ShopRoomVisualBinder`

## 8.5. Требования к расширяемости ShopRoom

Магазин должен без изменения базовой модели поддерживать:

- продажу не только артефактов, но и способностей, кубов, расходников, услуг;
- глобальные и локальные скидки;
- уникальные товары события;
- редкости и весовые таблицы выпадения;
- модификаторы от сложности, артефактов и событий;
- разные шаблоны layout магазина.

## 9. EventRoom

## 9.1. Назначение

`EventRoom` описывает комнату события, где основой взаимодействия являются текст события и варианты выбора.

Она отвечает за:

- ссылку на текстовый сценарий события;
- список вариантов действий;
- ссылки на обработчики исходов;
- правила доступности вариантов;
- визуальную и текстовую презентацию события.

## 9.2. Что должно находиться в EventRoom

### Обязательные данные
- `event_scenario_ref` — ссылка на сценарий события;
- `event_text_ref` — ссылка на текст/локализационный ключ;
- `choice_set_ref` — ссылка на набор вариантов;
- `choice_condition_refs` — ссылки на условия доступности;
- `choice_resolution_ref` — ссылка на резолвер исходов;
- `event_rule_refs` — ссылки на правила события;
- `event_visual_ref` — ссылка на визуальную конфигурацию.

### Runtime-данные
- какие варианты уже раскрыты;
- какие варианты заблокированы;
- выбрал ли игрок опцию;
- был ли исход уже зарезолвен;
- примененные последствия события.

## 9.3. Что не должно находиться в EventRoom

Не должно быть:

- захардкоженного текста прямо в логике комнаты как основного источника контента;
- функции `resolve_choice()` внутри комнаты;
- ручного кода выдачи наград/штрафов внутри комнаты;
- UI-логики диалогового окна внутри комнаты.

## 9.4. Внешние системы для EventRoom

Рекомендуемые внешние сервисы:

- `EventScenarioResolver`
- `EventChoiceAvailabilityPolicy`
- `EventChoiceResolver`
- `EventConsequenceApplier`
- `EventRoomVisualBinder`

## 9.5. Требования к расширяемости EventRoom

Событийная комната должна поддерживать:

- многошаговые события;
- условные ветви;
- скрытые варианты;
- варианты, зависящие от артефактов, статусов, куба карты, истории забега;
- уникальные редкие исходы;
- изменение не только ресурсов игрока, но и правил маршрута, магазина и будущих комнат.

## 10. BattleRoom

## 10.1. Назначение

`BattleRoom` описывает комнату боя и связывает маршрутную подсистему с боевой подсистемой.

Она отвечает за:

- ссылку на тип или шаблон монстров;
- ссылку на encounter-конфигурацию;
- правила конкретной боевой комнаты;
- визуал боевой арены;
- точки входа и выхода для запуска боевого сценария.

## 10.2. Что должно находиться в BattleRoom

### Обязательные данные
- `encounter_definition_ref` — ссылка на конфигурацию боя;
- `monster_pool_ref` — ссылка на таблицу или шаблон монстров;
- `encounter_generation_ref` — ссылка на генератор состава врагов;
- `battle_rule_refs` — ссылки на модификаторы боевой комнаты;
- `battle_start_ref` — ссылка на вход в боевой runtime;
- `battle_completion_ref` — ссылка на обработку завершения боя;
- `battle_visual_ref` — ссылка на визуальную конфигурацию арены.

### Runtime-данные
- сгенерированный `EncounterInstance`;
- статус боя (`not_started`, `in_progress`, `won`, `lost`);
- награды комнаты после боя;
- room-specific modifiers, влияющие на encounter.

## 10.3. Что не должно находиться в BattleRoom

Не должно быть:

- реализации пошагового боевого цикла;
- логики ИИ врагов;
- формул урона;
- визуальной логики анимаций;
- прямого хранения функций боя в комнате.

## 10.4. Внешние системы для BattleRoom

Рекомендуемые внешние сервисы:

- `BattleEncounterResolver`
- `BattleBootstrapper`
- `BattleCompletionHandler`
- `BattleRewardResolver`
- `BattleRoomPresentationBinder`

## 10.5. Требования к расширяемости BattleRoom

Боевая комната должна поддерживать:

- обычные бои;
- элитные бои;
- боссы;
- специальные модификаторы арены;
- scripted encounters;
- волны врагов;
- особые room rules, влияющие на старт боя, награды и ограничения.

## 11. Жизненный цикл комнаты

Ниже приведен обязательный целевой жизненный цикл.

### 11.1. Создание

`RoomFactory` или `RoomBuilder` создает комнату из `RoomDefinition` и связанных specialized definitions.

На этом этапе:
- назначается тип комнаты;
- подбираются refs;
- создается пустое runtime-состояние;
- подготавливаются object slots;
- назначается visual config.

### 11.2. Подготовка

Внешний `RoomLifecycleHandler` вызывает `prepare_ref`.

На этом этапе:
- генерируются runtime-объекты комнаты;
- применяются стартовые room rules;
- подготавливается визуальная конфигурация;
- фиксируются временные модификаторы от забега.

### 11.3. Вход

Когда игрок входит в комнату:
- публикуется событие входа;
- активируются entry-правила;
- presentation layer получает room visual config;
- запускается соответствующий сценарий комнаты.

### 11.4. Активная фаза

В зависимости от типа комнаты:
- магазин открывает ассортимент;
- событие показывает текст и варианты;
- бой запускает encounter.

### 11.5. Разрешение

Когда основной сценарий комнаты завершен:
- внешний handler исполняет `complete_ref`;
- награды/штрафы применяются к `RunState`;
- room state переводится в `completed`, `failed` или `abandoned`.

### 11.6. Выход

На выходе:
- публикуются события завершения;
- очищаются временные room-bound модификаторы;
- результат комнаты записывается в историю забега.

## 12. Интеграция с остальной архитектурой

## 12.1. Связь с RunState

Комната не должна владеть всем состоянием забега.

Она должна хранить только локальное состояние комнаты, а изменения более широкого масштаба применять через:

- `RunState`
- `RunContext`
- `RewardResolver`
- `RoomCompletionHandler`

## 12.2. Связь с событиями домена

Комнаты должны публиковать события высокого уровня, например:

- `RoomPrepared`
- `RoomEntered`
- `RoomActivated`
- `RoomCompleted`
- `RoomFailed`
- `ShopPurchaseCompleted`
- `EventChoiceSelected`
- `BattleRoomStarted`
- `BattleRoomResolved`

При этом публикация событий выполняется внешним event bus / room orchestrator, а не ad hoc кодом внутри каждой комнаты.

## 12.3. Связь с presentation layer

Presentation слой должен получать **RoomViewModel / RoomVisualConfig**, а не доступ к произвольной доменной логике.

Это нужно, чтобы:
- можно было менять сцены;
- можно было иметь текстовый и визуальный режимы представления;
- доменная модель комнаты не зависела от конкретного UI-фреймворка.

## 14. Правила эволюции подсистемы

При дальнейшем развитии архитектуры нужно соблюдать следующие правила.

### 14.1. Новые типы комнат добавляются без переписывания базового Room

При добавлении нового типа комнаты допустимо:
- добавить новый specialized definition;
- добавить новый подкласс данных;
- добавить новый handler/resolver;
- добавить новые `*_ref` поля.

Нежелательно:
- превращать базовый `Room` в god-object;
- добавлять в `Room` типоспецифичную логику исполнения;
- завязывать весь runtime на единый giant switch.

### 14.2. Подклассы остаются тонкими

Если подкласс начал содержать сложную прикладную логику, эту логику нужно выносить:

- в policy;
- в resolver;
- в отдельный service;
- в definition-driven сценарий.

### 14.3. Контент расширяется через definitions

Новый магазин, новый event template или новая encounter-комната должны по возможности добавляться через новые данные, а не через новый hardcoded класс.

### 14.4. Визуал не должен ломать домен

Изменение сцены, layout или темы комнаты не должно требовать переписывания доменного `Room`, если не изменился сам gameplay-контракт.

## 15. Минимальный итоговый контракт

Если свести архитектуру к минимальному обязательному правилу, то оно такое:

> `Room` и ее подклассы — это контейнеры типа комнаты, данных комнаты, runtime-состояния и ссылок на внешние обработчики.
> 
> Они не являются местом, где живет исполняемая логика магазина, события или боя.

Именно этому правилу должна следовать будущая реализация.
