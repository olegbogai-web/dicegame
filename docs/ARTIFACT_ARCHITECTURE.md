# Архитектура подсистемы артефактов

## Назначение
Документ фиксирует архитектурный стандарт для артефактов в рамках забега: хранение, редкость, получение, применение пассивных эффектов и расширяемость без переписывания ядра.

---

## Инварианты
- Артефакты — предметы, получаемые **во время забега**.
- У каждого артефакта обязательно есть:
  - `id` (стабильный ключ);
  - `name`;
  - `sprite`;
  - `rarity`.
- Редкости: `common` (обычное), `uncommon` (необычное), `rare` (редкое), `unique` (уникальное).
- Редкость влияет на частоты выпадения через таблицы веса.
- Артефакт `unique` может существовать в текущем забеге только в количестве `1`.
- Базовые артефакты хранятся в `PlayerBaseStat`.
- Все полученные в забеге артефакты хранятся в runtime-стате игрока (`Player`).

---

## Модель данных (что хранить)

## 1) Контент (definition)
`ArtifactDefinition` (ресурс/данные) хранит только описание артефакта:
- метаданные: `id`, `name`, `sprite`, `rarity`, `tags`;
- список пассивных модификаторов (`modifiers`);
- список триггерных реакций (`triggers`) — опционально.

**Не хранить в definition:**
- текущее состояние забега;
- ссылки на сцену боя/комнаты/UI;
- прямую прикладную логику экрана.

## 2) Runtime-инстанс
`ArtifactInstance` хранит только состояние конкретного экземпляра в забеге:
- ссылка на `ArtifactDefinition`;
- runtime-поля (если нужны): счетчики, флаги «сработал в этом ходу/бою» и т.д.

**Не хранить в instance:**
- правила выбора наград;
- визуальные ссылки UI-слоя;
- логику сохранения мета-прогрессии.

## 3) Контейнеры игрока
- `PlayerBaseStat.artifacts_base` — стартовые/постоянные базовые артефакты.
- `Player.artifacts_runtime` — артефакты, полученные в текущем забеге.

Итоговый активный набор артефактов игрока = `base + runtime` (через агрегатор, без копипасты логики в UI/комнатах/бою).

---

## Пайплайн получения и применения

## 1) Выдача артефакта
1. Система награды формирует пул кандидатов по контексту.
2. Применяет веса редкости.
3. Фильтрует `unique`: уже полученные в этом забеге исключаются.
4. Создает `ArtifactInstance` и добавляет в `Player.artifacts_runtime`.
5. Публикует событие `ArtifactGranted`.

## 2) Применение эффектов
Эффекты артефактов применяются только через общий pipeline модификаторов/триггеров:
- пассивные бонусы (например, макс HP, урон, шанс карт, сила состояний);
- реакции на события (`OnBattleStart`, `OnTurnStart`, `OnCardDropRoll`, и т.д.).

Запрещено применять эффекты «напрямую» из UI, комнаты или случайного контроллера.

---

## Распределение ответственности по файлам

## Что должно быть
- **Файлы definitions**: только декларация данных артефакта.
- **Файлы runtime артефактов**: только runtime-состояние экземпляров и контейнеров.
- **Файлы reward/loot**: только выбор и выдача артефактов (с учетом редкости/unique).
- **Файлы effect pipeline**: только расчет и применение модификаторов/триггеров.
- **Файлы UI/presentation**: только отображение (иконка, название, подсказка, редкость).

## Чего не должно быть
- В UI-файлах не должно быть расчета шанса выпадения или применения геймплейных эффектов.
- В файлах definitions не должно быть прямых вызовов runtime-логики.
- В combat/room-классах не должно быть «вшитых» проверок конкретных `artifact_id`.
- В одном «боже-объекте» не должно смешиваться: контент + дроп + эффекты + UI.

---

## Расширяемость (обязательные правила)
- Новый артефакт добавляется как новый `ArtifactDefinition`, без изменения ядра боя/комнат.
- Новый тип эффекта добавляется через расширение общего pipeline модификаторов/триггеров.
- Новая редкость добавляется через enum + weight table, без переписывания логики выдачи.
- Логика `unique` централизована в одном сервисе валидации выдачи.

---

## Минимальные события для интеграции
- `ArtifactGranted`
- `RunStarted`
- `BattleStarted`
- `TurnStarted`
- `DamageCalculated`
- `StatusApplyCalculated`
- `CardDropChanceCalculated`

Артефакты подписываются на события только через общий dispatcher/effect runtime.

### Контракт триггеров начала/конца хода
- Для триггеров `on_turn_start` и `on_turn_end` значение по умолчанию: событие считается относящимся к **ходу игрока**.
- Это означает, что артефактный триггер без дополнительных параметров срабатывает только когда `turn_owner = player`.
- Если нужно явно изменить область срабатывания, это настраивается data-driven в `ArtifactTriggerDefinition.parameters.turn_owner`:
  - `player` — только ход игрока (значение по умолчанию);
  - `monster` — только ходы монстров;
  - `any` — и ход игрока, и ходы монстров.
- Контракт обязателен для runtime-обработки и контентных `.tres`: поведение не задается хардкодом по `artifact_id`.

## Runtime-контракт эффекта "при получении"
- Для data-driven артефактов с мгновенным эффектом при выдаче используется `ArtifactDefinition.metadata`.
- Ключ `on_grant_max_hp_bonus` задает, на сколько артефакт увеличивает одновременно:
  - текущее HP игрока;
  - максимальное HP игрока в рамках текущего забега.
- Применение выполняется через единый метод выдачи артефакта (`Player.grant_artifact`), без проверок конкретных `artifact_id` в UI/бою/комнатах.

## Добавленные артефакты (контент)
- `Банан` (`artifact_id = banana`): при получении `+10` к текущему и максимальному HP (`metadata.on_grant_max_hp_bonus = 10`).
- `Яблоко` (`artifact_id = apple`): при получении `+20` к текущему и максимальному HP (`metadata.on_grant_max_hp_bonus = 20`).
- `Арбуз` (`artifact_id = watermelon`): при получении `+30` к текущему и максимальному HP (`metadata.on_grant_max_hp_bonus = 30`).


## Интеграция с post-battle reward (`artifact_+`)

В текущем runtime-потоке (`content/combat/reward/post_battle_reward_flow.gd`) выбор артефактов реализуется через функции:
- `_show_artifact_reward_options`;
- `_build_artifact_reward_options`;
- `_roll_artifact_reward_rarity`;
- `_pick_artifact_by_rarity_with_fallback`;
- `_select_artifact_reward`.

UI-часть использует 3D-шаблон `artefact_frame_reward` с дочерним `artefact_icon_reward` (сцена `new_battle_table`), где иконка подставляется из `ArtifactDefinition.sprite`.

## Границы документа
Документ **описывает**:
- модель артефактов (content/runtime);
- хранение base/runtime;
- редкость и правило unique;
- pipeline применения эффектов;
- ответственность файлов.

Документ **не описывает**:
- детальный UI/верстку экранов;
- балансные числа (проценты, точные веса, конкретные значения урона);
- конкретные реализации других подсистем (combat, rooms, events), кроме контрактов интеграции.

## Базовые артефакты-эталоны для проектирования

- `Ожерелье силы` (`artifact_id = strength_necklace`):
  - редкость: `common`;
  - тип эффекта: триггерный (`on_battle_start`);
  - действие триггера: применяет к игроку `1` стак статуса `Сила` через общий pipeline событий/эффектов;
  - реализация должна оставаться data-driven: эффект задается в `ArtifactDefinition.triggers`, а не в `Combat`/`BattleRoom` через проверки конкретного `artifact_id`.
- `Зуб древней змеи` (`artifact_id = ancient_snake_tooth`):
  - редкость: `unique`;
  - тип эффекта: триггерный (`on_turn_end`);
  - действие триггера: снимает у игрока до `2` стаков `Яда`, затем накладывает `Регенерацию` в размере `1` стака за каждый снятый стак;
  - реализован через data-driven trigger `effect_type = convert_status_on_remove` в `ArtifactRuntime`, без проверок конкретного `artifact_id` в боевых/room/UI классах.
