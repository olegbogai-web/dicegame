# Архитектура подсистемы Combat

## Статус документа

Этот документ является **специализированным архитектурным стандартом** для боевой подсистемы Dicegame.

Он обязателен для любых будущих изменений, связанных с:

- пошаговым боем;
- очередностью ходов;
- исполнением способностей;
- боевыми кубами;
- порядком активации игрока и монстров;
- завершением боя;
- интеграцией боя с боевыми комнатами.

Документ фиксирует целевую архитектуру так, чтобы боевая логика оставалась расширяемой, предсказуемой и data-driven.

Связанный документ по архитектуре решений монстров: [docs/MONSTER_AI_ARCHITECTURE.md](./MONSTER_AI_ARCHITECTURE.md). Он фиксирует отдельную фазовую модель monster turn, правило старта AI только после полной остановки кубов, общий policy конца хода и обязательное вынесение reusable execution/visual logic из `battle_table`.


Связанный документ по архитектуре состояний: [docs/STATE_ARCHITECTURE.md](./STATE_ARCHITECTURE.md). Он фиксирует контракт хранения статусов на сущностях, правила стэкинга, обязательные триггеры (включая конец хода) и границы между `Combat` и status runtime.

Ключевое актуальное состояние для боевого баланса: `bleeding` / «Кровотечение» — наносит урон в начале хода владельца и затухает на 1 стак в конце хода владельца.

Связанный документ по post-battle кубам награды и денег: [docs/REWARD_AND_MONEY_DICE_ARCHITECTURE.md](./REWARD_AND_MONEY_DICE_ARCHITECTURE.md). Он фиксирует, какие кубы хранятся в базовых/рантайм статах игрока и как они бросаются после победы в бою.

Связанный документ по архитектуре улучшения карт: [docs/CARD_UPGRADE_ARCHITECTURE.md](./CARD_UPGRADE_ARCHITECTURE.md). Он фиксирует инвариант, что upgraded-способности игрока и монстров исполняются тем же combat runtime без специальных веток по типу версии.

---

## 1. Цель подсистемы Combat

Подсистема `Combat` отвечает за исполнение законченного боевого сценария между игроком и одной или несколькими вражескими сущностями.

Ее задача — управлять:

- созданием боевого состояния;
- порядком ходов;
- броском и жизненным циклом кубов хода;
- расходом кубов на способности;
- резолвом способностей и эффектов;
- проверкой условий победы и поражения;
- публикацией боевых событий;
- возвратом итогового результата в систему комнат и забега.

Подсистема боя является **доменным исполнителем правил боя**, но не должна смешивать в себе:

- UI-логику экрана боя;
- layout боевой сцены;
- визуальную анимацию броска;
- конкретные room-сценарии;
- hardcode логики отдельных монстров в ядре очередности.

## 2. Базовые принципы

### 2.1. Бой пошаговый и детерминированный по фазам

Бой состоит из повторяющихся ходов сущностей.

Базовый порядок:

1. первым всегда ходит игрок;
2. затем ходят монстры;
3. если монстров несколько, они ходят последовательно;
4. раньше ходит тот монстр, у которого больше кубов;
5. после завершения всех ходов монстров начинается следующий ход игрока.

Если у нескольких монстров одинаковое число кубов, архитектура должна поддерживать отдельную политику tie-break, чтобы правило разрешения было явным и расширяемым. По умолчанию рекомендуется использовать стабильный детерминированный порядок по позиции в encounter или по `spawn_index`.

### 2.2. Победа и поражение завершают бой мгновенно

Бой обязан завершаться немедленно, без ожидания следующей фазы, если выполнено хотя бы одно условие:

- все монстры мертвы — результат `player_victory`;
- игрок мертв — результат `player_defeat`.

Проверка этих условий должна выполняться после каждого эффекта, который потенциально меняет состояние жизни существа.

### 2.3. Кубы — ресурс хода существа

Каждый ход существа начинается с броска всех доступных кубов этого существа.

Дальше сущность:

- получает `rolled dice pool`;
- может перераспределять / перетаскивать кубы в допустимые слоты стоимости;
- тратит кубы на активацию способностей;
- теряет потраченные кубы до следующего собственного хода;
- завершает ход добровольно или автоматически, если доступных действий больше нет.

### 2.4. Бой должен быть data-driven

Ядро боя не должно содержать жестко зашитых правил вроде:

- конкретная способность игрока делает X;
- конкретный монстр всегда бьет на Y;
- отдельный UI-поток вручную меняет HP в обход резолвера.

Вместо этого ядро должно исполнять:

- `AbilityDefinition`;
- правила целей (`AbilityTargetRule`);
- правила стоимости (`AbilityCost`);
- правила условий (`AbilityCondition`, `AbilityDiceCondition`);
- боевые эффекты;
- policy-объекты порядка хода и завершения боя.

### 2.5. Краткий реестр monster-only способностей

Чтобы при следующих изменениях не терялась семантика контента, monster-only способности фиксируются в текстовой архитектурной документации.

Текущее дополнение:

- `rat_bite` / «Укус крысы» (`content/abilities/definitions/rat_bite.tres`):
  - owner scope: только монстр (`owner_scope = MONSTER`), игрок получить не может через reward/shop flow;
  - стоимость: 1 любой боевой куб;
  - эффект: `damage = value(selected_die)` (урон равен значению выбранного куба);
  - flavor: «Лучше быть осторожнее с этими зубами».
- `goblin_inflict_poison` / «Нанести яд» (`content/abilities/definitions/goblin_inflict_poison.tres`):
  - owner scope: только монстр (`owner_scope = MONSTER`);
  - стоимость: 1 любой боевой куб;
  - эффект: накладывает на самого монстра `poisoner` (1), усиливая будущие наложения `poison`;
  - flavor: «Лучше не знать, откуда они берут свой «яд» прямо во время боя».
- `goblin_poison_strike` / «Отравляющий удар» (`content/abilities/definitions/goblin_poison_strike.tres`):
  - owner scope: только монстр (`owner_scope = MONSTER`);
  - стоимость: ровно 2 куба со значением 4+;
  - эффект: наносит 2 урона и накладывает `poison` (2) на цель;
  - flavor: «Палка, смазанная говном. Просто и эффективно».

### 2.6. Краткий реестр utility-способностей игрока

- `reroll` / «Переброс кубика» (`content/abilities/definitions/reroll.tres`):
  - owner scope: игрок (`owner_scope = PLAYER`), доступна в player reward/shop flow;
  - стоимость: base не требует кубов;
  - таргет: выбранный боевой куб игрока;
  - эффект: перебрасывает выбранный куб (в ветке `1.2` дополнительно лечит владельца на значение переброшенного куба);
  - лимит использования: `cooldown_turns = 1` (фактически 1 раз за собственный ход, откат в начале следующего хода владельца);
  - ветка `1` (`reroll_upgrade_1`) — до `2` использований за ход;
    - `1.1` (`reroll_upgrade_1_1`) — до `3` использований за ход;
    - `1.2` (`reroll_upgrade_1_2`) — до `2` использований за ход + лечение от результата переброса;
  - ветка `2` (`reroll_upgrade_2`) — без отката, но требует `1` любой куб на оплату;
    - `2.1` (`reroll_upgrade_2_1`) — как ветка `2` + `Регенерация 2` владельцу;
    - `2.2` (`reroll_upgrade_2_2`) — как ветка `2` + снятие `1` стака случайного отрицательного статуса;
  - runtime note: поддержаны data-driven effect types `reroll_dice_heal_owner_by_result` и `cleanse_random_negative_status` с debug-логированием;
  - редкость: `UNCOMMON`.
- `reroll_all` / «Переброс всего» (`content/abilities/definitions/reroll_all*.tres`):
  - owner scope: игрок (`owner_scope = PLAYER`), доступна в player reward/shop flow;
  - стоимость: не требует кубов;
  - таргет: глобальный (после выбора способности можно нажать в любую точку экрана);
  - base-эффект: перебрасывает все оставшиеся кубы игрока, которые еще не были потрачены в текущей активации;
  - base-лимит использования: `once_per_battle = true` (строго 1 раз за бой);
  - ветка `1` (`reroll_all_upgrade_1`) — `2` использования за бой, не чаще `1` раза за ход;
    - `1.1` (`reroll_all_upgrade_1_1`) — `3` использования за бой, не чаще `1` раза за ход;
    - `1.2` (`reroll_all_upgrade_1_2`) — `2` использования за бой без ограничения «1 раз за ход»;
  - ветка `2` (`reroll_all_upgrade_2`) — base-переброс + создание копии случайного куба с доски;
    - `2.1` (`reroll_all_upgrade_2_1`) — создает две копии случайного куба;
    - `2.2` (`reroll_all_upgrade_2_2`) — не перебрасывает кубы со значением `>= 5`;
  - runtime note: поддержаны data-driven параметры `max_uses_per_battle`, `max_uses_per_turn`, `copies_count`, `skip_reroll_if_value_gte`;
  - редкость: `RARE`.
- `death_blow`-линейка / «Смертельный удар» (`content/abilities/definitions/death_blow*.tres`):
  - owner scope: игрок (`owner_scope = PLAYER`);
  - ветка `1.x`: вариации от одиночного куба (`6`, либо `5/6`) с уроном `15..25`; `1.2` дополнительно лечит игрока на `15`, если цель убита этим ударом;
  - ветка `2.x`: вариации от `2x6`/`3x6` с уроном `50..75`; `2.2` дополнительно накладывает `disarmed` (1) на цель;
  - runtime note: условное лечение при убийстве реализуется через generic effect type `heal_if_target_dead` в `BattleEffectRuntime`, без ветки под конкретный `ability_id`.
- `needle_throw`-линейка / «Выстрел из мушкета» (`content/abilities/definitions/needle_throw*.tres`):
  - owner scope: игрок (`owner_scope = PLAYER`), редкость: `RARE`;
  - base: 1 любой куб, урон `floor(1.5 * value(selected_die))`;
  - ветка `1.x`: эскалация урона `2.25x -> 3x -> 5x`; у `1.2` есть `cooldown_turns = 1` (эффективно 1 использование за собственный ход);
  - ветка `2.x`: ужесточение dice-условий (`<=3`, затем `1..2`) в обмен на более высокий множитель урона (`4x`, `8x`), альтернативная `2.2` дает `5x` при условии `<=3`.
- `strengthening`-линейка / «Усиление» (`content/abilities/definitions/strengthening*.tres`):
  - owner scope: игрок (`owner_scope = PLAYER`), редкость: `COMMON`;
  - base: до 3 кубов, сумма `>= 9`, накладывает `Сила +3`;
  - ветка `1.x`: порог снижается (`>=7`, затем `>=6`) или повышается эффект до `Сила +4` в варианте `1.2`;
  - ветка `2.x`: строгие суммы (`ровно 9`, `ровно 12`) и высокий бонус `Сила +5/+6/+9`;
  - ветка `3.x`: эскалация `Сила +4 -> +5`, альтернативная `3.2` ограничивает оплату максимум 2 кубами при `Сила +6`.
- `poison_injection`-линейка / «Отравляющий укол» (`content/abilities/definitions/poison_injection*.tres`):
  - owner scope: игрок (`owner_scope = PLAYER`), редкость: `COMMON`;
  - base: требует ровно 1 куб со значением `1`, накладывает `poison` (`2`) на выбранную цель;
  - ветка `1.x`: усиливает стэкинг `poison` (`3 -> 4`), вариант `1.2` меняет стоимость на `2x1` и добавляет `poisoner 1` владельцу;
  - ветка `2.x`: ослабляет требование по оплате (`1..2`, `1..3`, затем любой куб) ценой снижения/фиксации стэков (`2`, `2`, `1`);
  - runtime note: использует стандартный `apply_status` pipeline (включая self-target через `parameters.target = \"source\"`), дополнительного хардкода в `BattleEffectRuntime` не требует.

### 2.7. Реестр хардкод-кубов в Combat runtime

- `perevertysh` / «Перевертыш» (`content/dice/definitions/perevertysh.tres`):
  - trigger: ПКМ по кубу на поле во время хода игрока;
  - effect: локальный переброс только этого куба;
  - limit: 1 раз за ход игрока;
  - runtime hooks: `PlayerAbilityInputController::PEREVERTYSH_DICE_NAME`, `PlayerAbilityInputController::_try_reroll_perevertysh_die`, `battle_table::_perevertysh_reroll_turn_counter`.
- `kamikaze` / «Камикадзе» (`content/dice/definitions/kamikaze.tres`):
  - trigger: способность активирована и в `consumed_dice` есть куб `kamikaze`;
  - effect: переброс всех оставшихся кубов игрока (кроме уже потраченных в этой активации);
  - runtime hooks: `BattleEffectRuntime::KAMIKAZE_DICE_NAME`, `BattleEffectRuntime::_has_consumed_dice_with_name`, `BattleEffectRuntime::_reroll_remaining_player_dice`.
- `duplicate` / «Дубликат» (`content/dice/definitions/duplicate.tres`):
  - trigger: любой бросок этого куба (обычный старт хода, `reroll`, `reroll_all`, хардкод-перебросы);
  - effect: вместо одного инстанса в runtime создаются два одинаковых куба;
  - runtime hooks: `BoardController::DUPLICATE_DICE_NAME`, `BoardController::_expand_duplicate_requests`, `BoardController::_is_duplicate_dice_request`.
- `joker`-линейка (`content/dice/definitions/joker.tres`, `content/dice/definitions/almost_real_joker.tres`, `content/dice/definitions/false_joker.tres`):
  - trigger: в любой ячейке выбранной способности стоит куб, у которого верхняя грань имеет `text_value = "joker"`;
  - effect: активация способности разрешается независимо от проверок dice-условий (value/face/tags/use_conditions);
  - runtime hooks:
    - `BattleAbilityRuntime::JOKER_FACE_ID`;
    - `BattleAbilityRuntime::_has_joker_override_dice`;
    - `PlayerAbilityInputController::JOKER_FACE_ID`;
    - `PlayerAbilityInputController::_has_joker_override_in_consumed_dice`;
    - `PlayerAbilityInputController::_dice_matches_slot` (джокер можно поставить в любую ячейку способности).
- `prilipala` / «Прилипала» (`content/dice/definitions/prilipala.tres`):
  - trigger: куб участвует в оплате способности;
  - effect: на первой активации остается в слоте и не удаляется, удаляется на второй активации;
  - runtime hooks: `BattleActivationAnimationRuntime::ABILITY_ACTIVATION_CONSUME_COUNTER_META`, `BattleActivationAnimationRuntime::_should_consume_dice_on_current_activation`.
- `golden` / «Золотой» (`content/dice/definitions/golden.tres`):
  - trigger: куб включен в бросок начала хода игрока;
  - effect: за каждый брошенный `golden` игрок получает `+1` монету;
  - runtime hooks: `BattleTurnOrchestrator::GOLDEN_DICE_NAME`, `BattleTurnOrchestrator::_apply_player_throw_coin_bonus`.
- `poisoned` / «Отравленный» (`content/dice/definitions/poisoned.tres`):
  - trigger: куб входит в `consumed_dice` при активации способности;
  - effect: накладывает на владельца `poison` в размере `5` стаков за каждый использованный куб;
  - доп. свойство data-уровня: `ability_activations_before_consume = 1000` (куб практически не расходуется);
  - runtime hooks: `BattleEffectRuntime::POISONED_DICE_NAME`, `BattleEffectRuntime::_apply_poisoned_dice_self_poison`.

## 3. Границы ответственности Combat

### Combat отвечает за

- создание `BattleState` / `CombatState`;
- регистрацию участников боя;
- определение текущего активного существа;
- расчет очереди ходов;
- бросок кубов начала хода;
- учет доступных, занятых, потраченных и удаленных до следующего хода кубов;
- валидацию возможности активации способности;
- применение эффектов способности;
- публикацию боевых событий;
- проверку завершения боя;
- формирование результата боя для комнаты;
- публикацию событий, по которым Monster AI узнает, что все кубы активного монстра остановились и decision phase может начаться.

### Combat не отвечает за

- выбор комнаты на карте;
- выдачу наград после боя;
- отображение drag-and-drop в UI;
- анимации полета куба, удара, лечения;
- загрузку сцены комнаты;
- внешний нарратив конкретной комнаты.

## 4. Целевая модель доменных сущностей

## 4.1. BattleState

Главный runtime-объект боя.

Содержит:

- идентификатор боя;
- ссылку на `BattleRoom` / encounter source;
- список участников;
- разбиение участников по сторонам (`player`, `enemy`);
- состояние очереди ходов;
- текущую фазу боя;
- текущий активный ход;
- журнал боевых событий;
- флаги завершения;
- итоговый результат боя.

`BattleState` не должен содержать UI-узлы и не должен напрямую знать о визуальных контролах.

## 4.2. Combatant

Унифицированная модель участника боя.

Подходит и для игрока, и для монстра.

Минимальные поля:

- `combatant_id`;
- `side` (`player` / `enemy`);
- ссылка на базовую сущность / definition;
- текущие HP, броню, статусы;
- набор способностей;
- профиль кубов;
- `dice_count` или эквивалентный runtime-параметр;
- флаг `is_alive`;
- metadata для порядка хода и таргетинга.

## 4.3. TurnState

Описывает текущий ход конкретного существа.

Содержит:

- `turn_owner_id`;
- номер раунда / номер хода;
- брошенные кубы;
- доступные кубы;
- кубы, размещенные в слотах стоимости;
- кубы, уже потраченные в этом ходу;
- список активированных способностей;
- флаг возможности завершить ход;
- причину завершения хода.

## 4.4. DicePool / TurnDicePool

Изолированный runtime-объект управления кубами внутри хода.

Нужен для разделения состояний кубов:

- `rolled` — только что брошен;
- `available` — может быть использован;
- `slotted` — помещен в слот стоимости;
- `spent` — потрачен на активацию способности;
- `consumed_until_next_turn` — удален до следующего хода владельца;
- `locked` / `disabled` — временно недоступен по эффекту.

Важно: потраченный куб не уничтожается навсегда, а лишь выходит из пула до следующего хода этого же существа, если иное не указано отдельным эффектом.

## 4.5. BattleResult

Итог боя, который возвращается наружу.

Минимум:

- `player_victory` / `player_defeat` / `cancelled`;
- причина завершения;
- список выживших;
- список погибших;
- сведения для post-battle resolution.

## 5. Фазовая модель боя

Рекомендуется фиксированная фазовая модель.

### 5.1. Battle setup

На этой фазе система:

- создает `BattleState`;
- регистрирует игрока;
- создает монстров encounter;
- сортирует стартовый вражеский порядок по числу кубов;
- подготавливает первый ход игрока.

### 5.2. Turn start

На старте хода активного существа система:

- выбирает владельца хода;
- создает новый `TurnState`;
- бросает все кубы существа;
- публикует событие начала хода;
- переводит бой в фазу принятия решений.

Примечание по артефактам: триггеры `on_turn_start` без явного `parameters.turn_owner` считаются привязанными к началу **хода игрока**; для срабатывания на ходах монстров требуется явная настройка (`monster` или `any`) в данных артефакта.

### 5.3. Decision / activation

На этой фазе активное существо:

- размещает кубы в слоты стоимости;
- выбирает способность;
- выбирает цель;
- инициирует `ActionIntent`;
- получает результат через `ActionResolver`.

Для ИИ здесь же исполняется выбор действия через внешний decision policy.

Подробные требования к границам monster-specific decision logic см. в [docs/MONSTER_AI_ARCHITECTURE.md](./MONSTER_AI_ARCHITECTURE.md).

### 5.4. Resolution

На этой фазе:

- проверяются условия способности;
- резервируются и списываются кубы;
- применяются эффекты;
- пересчитываются HP, броня, статусы и прочие runtime-показатели;
- после каждого значимого шага проверяются условия победы/поражения.

### 5.5. Turn end

На этой фазе:

- оставшиеся временные объекты хода очищаются;
- неиспользованные кубы завершают жизненный цикл хода;
- публикуется событие окончания хода;
- выбирается следующий участник;
- либо начинается следующий ход, либо бой завершается.

Примечание по артефактам: триггеры `on_turn_end` по умолчанию также исполняются только для хода игрока, пока в триггере не задан иной `parameters.turn_owner`.

### 5.6. Battle finished

На этой фазе:

- результат фиксируется в `BattleResult`;
- публикуется событие конца боя;
- управление возвращается в боевую комнату / run flow.

## 6. Порядок ходов

## 6.1. Базовое правило

Последовательность хода внутри живого encounter:

- игрок;
- монстр с наибольшим числом кубов;
- следующий монстр по убыванию числа кубов;
- ...;
- после последнего живого монстра снова игрок.

## 6.2. Динамический пересчет порядка

Архитектура должна поддерживать два уровня порядка:

1. **базовый order snapshot на цикл врагов**;
2. **политику пересчета**, если количество кубов монстра изменилось во время боя.

Рекомендуемый стандарт:

- на начало каждой вражеской подфазы формируется список всех живых монстров;
- список сортируется по текущему числу кубов по убыванию;
- этот список отрабатывается последовательно;
- после завершения вражеской подфазы начинается новый ход игрока.

Такой подход делает правило понятным и естественно поддерживает эффекты, меняющие число кубов монстров между циклами.

## 6.3. Пропуск мертвых участников

Если участник умер до своего запланированного хода, он просто исключается из очереди без создания пустого хода.

## 6.4. Прерывание очереди завершением боя

Если в середине вражеской последовательности:

- умирает последний живой монстр;
- умирает игрок;

очередь немедленно останавливается, а бой завершается без доигрывания оставшихся ходов.

---

## 7. Структура хода существа

Каждый ход должен исполняться в одной и той же последовательности.

### Шаг 1. Бросок всех кубов существа

Система бросает все кубы, доступные этому существу на текущий ход.

Для игрока в этот список попадают только кубы с `DiceDefinition.scope = COMBAT` из `player.dice_loadout`. Это включает контентные кубы типа «Счастливая единица/двойка/.../шестерка»: при получении они добавляются в `dice_loadout` и начинают бросаться каждый ход в боевой комнате.

### Шаг 2. Формирование пула хода

Результаты броска переходят в пул доступных кубов.

### Шаг 3. Размещение кубов для оплаты

Существо может перемещать кубы в cost-слоты способностей, если это допускается правилами стоимости.

### Шаг 4. Активация способности

После выполнения условий стоимости существо активирует способность.

### Шаг 5. Списание кубов

Каждый куб, потраченный на активацию способности:

- удаляется из доступного пула текущего хода;
- переводится в состояние `consumed_until_next_turn`;
- возвращается только в начале следующего хода того же существа.

Исключение: если у куба задано `DiceDefinition.ability_activations_before_consume > 1`, фактическое удаление/перевод в consumed-состояние происходит только после достижения нужного числа активаций (например, `prilipala` удаляется со второго списания).

### Шаг 6. Выбор следующих действий

Существо может:

- активировать следующую способность;
- закончить ход;
- быть принудительно завершено системой, если действий не осталось.

### Шаг 7. Завершение хода

Ход закрывается и управление переходит следующему участнику очереди.

---

## 8. Контракт активации способности игроком

Игрок активирует способность через единый сценарий ввода.

### 8.1. Последовательность пользовательского действия

Для способности, требующей кубов:

1. игрок переносит нужное количество кубов в слоты стоимости;
2. игрок нажимает на способность;
3. игрок нажимает на цель способности.

Для способности без стоимости в кубах:

1. игрок нажимает на способность;
2. игрок нажимает на цель способности, если цель требуется.

### 8.2. Типы целей

Архитектура таргетинга должна поддерживать как минимум следующие варианты:

- на себя;
- на конкретного врага;
- на любого противника;
- на всех врагов;
- в любое место / глобально;
- на конкретный куб.

### 8.3. Разделение UI и домена

UI может реализовывать drag-and-drop, подсветку и выбор цели любым удобным способом, но доменная подсистема должна получать уже нормализованную структуру вроде:

- `selected_ability_id`;
- `source_combatant_id`;
- `selected_dice_ids`;
- `target_descriptor`.

Это позволяет не привязывать ядро боя к конкретной реализации интерфейса.

---

## 9. Модель активации способности

Рекомендуется следующий pipeline.

1. `AbilitySelection` — способность выбрана.
2. `CostAssignment` — кубы помещены в стоимость.
3. `TargetSelection` — цель зафиксирована.
4. `ActionIntent` — сформировано намерение.
5. `ActionValidation` — проверены право хода, стоимость, условия, валидность цели.
6. `CostCommit` — кубы окончательно списаны.
7. `EffectResolution` — применены эффекты.
8. `Aftermath` — проверены смерти, победа/поражение, реактивные события.

Если валидация не пройдена, ядро не должно частично применять последствия.

---

## 10. Монстры и ИИ

## 10.1. Монстр использует тот же боевой контракт

Монстры должны жить в той же модели боя, что и игрок:

- имеют свои кубы;
- бросают все кубы в начале своего хода;
- активируют способности;
- тратят кубы;
- завершают ход.

## 10.2. ИИ не должен быть встроен в BattleState

Выбор действия монстра должен быть вынесен в отдельный компонент, например:

- `EnemyDecisionPolicy`;
- `MonsterTurnPlanner`;
- `AbilityPriorityEvaluator`.

`BattleState` хранит только данные, а решение выбирает внешний сервис.

## 10.3. Последовательность хода монстра

На своем ходу монстр:

1. получает бросок всех своих кубов;
2. оценивает доступные способности;
3. выбирает допустимое действие или завершает ход;
4. тратит кубы;
5. применяет эффект;
6. либо повторяет цикл выбора, либо завершает ход.

---

## 11. События боя

Бой должен публиковать события, пригодные для реактивных систем и визуализации.

Минимальный набор:

- `battle_started`;
- `turn_started`;
- `dice_rolled`;
- `dice_slotted`;
- `ability_selected`;
- `ability_activated`;
- `cost_paid`;
- `effect_applied`;
- `damage_applied`;
- `heal_applied`;
- `combatant_died`;
- `turn_ended`;
- `battle_finished`.

События нужны для:

- статусов и артефактов-реакций;
- логирования;
- анимаций;
- аналитики;
- обучающих подсказок.

---

## 12. Интеграция с боевой комнатой

Боевая комната не должна подменять правила боя.

Правильная схема зависимости:

- `BattleRoom` поставляет encounter setup;
- `Combat` исполняет бой;
- `BattleRoom` принимает `BattleResult`;
- затем room/runtime/run системы обрабатывают награды, завершение комнаты и переход дальше.

Это означает, что:

- логика очередности живет в `Combat`;
- логика трат кубов живет в `Combat`;
- логика победы/поражения живет в `Combat`;
- логика room lifecycle остается в подсистеме `Room`.

---

## 13. Правила расширения

Допустимо расширять боевую подсистему через:

- новые `AbilityDefinition`;
- новые типы эффектов;
- новые target rules;
- новые условия стоимости;
- новые policy порядка хода;
- новые AI-политики;
- новые реакции на события.

Нельзя расширять бой через:

- копирование отдельной версии turn loop для каждого монстра;
- дублирование player flow и enemy flow в двух разных несвязанных системах;
- прямое изменение HP/статусов из UI в обход резолвера;
- хранение room-specific правил победы внутри общего `BattleState`, если они не оформлены как policy.

---

## 14. Минимальный архитектурный инвариант

Любая будущая реализация боя должна сохранять следующие инварианты:

- игрок всегда начинает бой первым;
- монстры ходят после игрока;
- монстры ходят последовательно в порядке убывания числа кубов;
- каждый ход начинается с броска всех кубов активного существа;
- куб, потраченный на способность, недоступен до следующего хода владельца;
- куб может иметь отложенное удаление по `DiceDefinition.ability_activations_before_consume` (например, `prilipala`);
- способность исполняется только после валидного выбора стоимости и цели;
- смерть игрока или уничтожение всех монстров немедленно завершает бой;
- комната не дублирует боевую логику, а использует результат `Combat`.

Именно эти инварианты являются ядром текущего дизайна боя.

## 16. Обновление: runtime-поддержка составного набора кубов монстра (март 2026)

Для отказа от хардкода количества однотипных кубов добавлена поддержка `dice_loadout` у монстров.

Новые/обновленные функции runtime-слоя:
- `MonsterDefinition.get_combat_dice_loadout()` — возвращает валидный combat-набор кубов монстра;
- `MonsterDefinition.get_combat_dice_count()` — возвращает размер `dice_loadout` (или fallback в `dice_count`);
- `BattleTurnOrchestrator.throw_current_turn_dice(...)` — при ходе монстра бросает кубы из `monster_view.dice_loadout`, fallback — legacy `dice_count`.

Также в `BattleAbilityRuntime` добавлены функции для корректного подбора стоимости с ограничением по сумме:
- `_has_total_value_constraint(dice_condition)`;
- `_collect_dice_with_total_value(...)`;
- `_collect_dice_with_total_value_backtrack(...)`.

Это устраняет класс ошибок, когда greedy-подбор не находил валидную комбинацию кубов при наличии ограничения по total value.

## 17. Обновление: визуальный размер монстров (апрель 2026)

Для data-driven управления визуалом монстров в боевой комнате в `MonsterDefinition` добавлен параметр:

- `size_multiplier` (`0.2..2.0`, default `1.0`).

Назначение параметра:
- масштабировать спрайт монстра в `BattleSceneViewRenderer`;
- масштабировать вместе со спрайтом HP-frame/HP-bar монстра;
- сдвигать стартовую точку статус-иконок так, чтобы они начинались под левым краем изображения при любом размере;
- **не** масштабировать сами статус-иконки (меняется только их позиционирование относительно спрайта).

Runtime-интеграция:
- `BattleRoom.set_monsters_from_definitions(...)` переносит `size_multiplier` в `CombatantViewData` и применяет его к `base_scale` монстра;
- `BattleSceneViewRenderer._apply_statuses_to_sprite(...)` учитывает `size_multiplier` для enemy-side и компенсирует масштаб статус-иконок обратным коэффициентом.
