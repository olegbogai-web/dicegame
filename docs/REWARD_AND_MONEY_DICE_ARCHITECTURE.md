# Архитектура куба награды и куба денег

## Статус документа

Этот документ задает архитектурный стандарт для двух run-кубов:

- `reward dice` (куб награды);
- `money dice` (куб денег).

Документ обязателен для изменений в `player base stats`, `player runtime stats`, пост-бойовой выдаче наград и связанных сервисах.

Связанный документ по архитектуре улучшения карт: [docs/CARD_UPGRADE_ARCHITECTURE.md](./CARD_UPGRADE_ARCHITECTURE.md). Он фиксирует модель ветвящихся upgrade-вариантов способности, контракт сохранения общего `ability_id` и правило переиспользования `ability_reward` flow для грани `card_up`.

---

## 1. Обязательные правила модели кубов

1. Куб награды и куб денег являются отдельными сущностями, но оба **наследуют базовый `Cube`**.
2. Оба куба хранятся:
   - в базовых статах игрока (стартовая конфигурация забега);
   - в runtime-статах игрока (текущее состояние в рамках одного забега).
3. Оба куба могут изменяться во время забега, поэтому характеристики граней не считаются константными.
4. Любые изменения граней/весов/модификаторов применяются к runtime-экземплярам, не к глобальному shared-ресурсу.

---

## 2. Изменения в статах игрока

## 2.1. Базовые статы (`PlayerBaseStat`)

В базовых статах должны быть отдельные поля:

- `base_reward_cube: CubeDefinition` — стартовый куб награды;
- `base_money_cube: CubeDefinition` — стартовый куб денег.

Базовые поля определяют только старт забега. Дальнейшие модификации идут в runtime.

## 2.2. Рантайм статы (`Player`)

При создании/сбросе рантайм-игрока:

- `runtime_reward_cube` инициализируется копией `base_reward_cube`;
- `runtime_money_cube` инициализируется копией `base_money_cube`.
- `runtime_reward_cubes` и `runtime_money_cubes` инициализируются как коллекции runtime-кубов для соответствующих scope;
- добавлены отдельные коллекции по scope (`dice_loadout`, `runtime_cube_global_map`, `runtime_reward_cubes`, `runtime_money_cubes`, `runtime_event_cubes`) и API `Player.grant_runtime_cube` / `Player.get_runtime_cubes_by_scope`, чтобы не смешивать кубы разных подсистем в рантайме.

Ключевое правило: в течение текущего забега runtime-наборы кубов могут меняться и накапливаться (дубликаты не-unique разрешены), при этом `unique`-кубы не дублируются.

---

## 3. Когда бросаются кубы

После победы в бою выполняется post-battle reward phase:

1. бросается `reward dice`;
2. одновременно с ним бросается `money dice`;
3. если на `reward dice` выпала грань `монеты`, выполняется **дополнительный физический бросок** `money dice` из `player.runtime_money_cubes`;
4. начисление монет происходит только после остановки всех соответствующих money-кубов (базовый post-battle бросок + дополнительный бросок по `money`);
5. для `money`-грани, после начисления монет, переход на глобальную карту выполняется с задержкой `2.0` секунды.

Это правило относится только к триггеру броска и не задает подробности экономики.

---

## 4. Базовая конфигурация куба награды

Базовый `reward dice` имеет 6 граней:

1. `артефакт` — показать выбор 1 из 3 случайных артефактов случайной редкости;
2. `карта` — показать выбор 1 из 3 случайных карт случайной редкости;
3. `куб` — показать выбор 1 из 3 случайных кубов случайной редкости;
4. `улучшение куба` — показать выбор 1 из 3 случайных улучшений для случайного куба случайной редкости;
5. `улучшение карты` — показать выбор, какую из 3 случайных имеющихся карт улучшить;
6. `монеты` — бросить дополнительный `money dice`.

В этом документе не фиксируются таблицы контента, вероятности редкости и конкретные реализации наград.

---


## 4.1. Правила редкости для наград

Для способностей и кубов используется единая шкала редкости:

- `обычное` (базовое значение);
- `необычное`;
- `редкое`;
- `уникальное`.

Редкость влияет на частоты выпадения в reward-таблицах.

Отдельный инвариант: награда редкости `уникальное` может быть выдана в рамках одного забега только один раз (независимо от того, способность это или куб).

Точные веса/формулы выпадения и конкретный runtime-алгоритм здесь не фиксируются и реализуются в отдельном блоке подсистемы наград.

## 4.2. Runtime-алгоритм награды способностями (`card_+`)

Если после победы на `reward dice` выпадает грань `card_+`:

1. генерируются 3 варианта способности для выбора игроком;
2. для каждого слота сначала бросается редкость (веса: `обычное 50`, `необычное 30`, `редкое 20`, `уникальное 10`);
3. затем случайно выбирается способность нужной редкости;
4. если в этой редкости нет валидных способностей, выполняется fallback к предыдущей редкости ниже (`уникальное -> редкое -> необычное -> обычное`);
5. способность, уже имеющаяся у игрока, не может попасть в выдачу;
6. внутри одной выдачи из 3 вариантов дубликаты также запрещены;
7. игрок выбирает одну карточку награды (`ability_reward`), и выбранная способность добавляется в `player.ability_loadout`;
8. новая способность становится доступной начиная со следующего боя.

Короткая актуализация текущего поведения:
- runtime-loadout способностей игрока ограничен `MAX_PLAYER_ABILITIES = 5`;
- если при `card_+` игрок уже имеет 5 способностей, выбранная новая способность заменяет случайную текущую способность в `ability_loadout` (вместо расширения колоды).

Этот алгоритм должен оставаться data-driven и брать пул способностей из контентных ресурсов, а не из жестко зашитого списка в коде.

Архитектура грани `card_up` (улучшение существующей способности) вынесена отдельно в [docs/CARD_UPGRADE_ARCHITECTURE.md](./CARD_UPGRADE_ARCHITECTURE.md), чтобы не смешивать flow новой карты (`card_+`) и flow улучшения (`card_up`) в одном стандарте.

### Текущие runtime-функции (presentation + orchestration)

В текущей реализации (`scenes/battle_table.gd`) за flow `card_+` отвечают:

- `_try_resolve_post_battle_reward_dice_result` — ожидает остановку post-battle кубов и разбирает грань куба награды;
- `_show_ability_reward_options` — запускает генерацию и показ 3 карточек награды;
- `_build_ability_reward_options` — формирует набор вариантов на основе редкости;
- `_roll_reward_rarity` — бросок редкости по весам;
- `_pick_ability_by_rarity_with_fallback` — выбор способности с fallback по редкостям вниз;
- `_resolve_base_frame_by_rarity` — выбор ассета рамки карточки по редкости;
- `_render_ability_reward_cards`, `_apply_reward_card_visual` и `_apply_reward_card_base_frame_visual` — визуализация карточек в `new_battle_table`;
- `_resolve_ability_reward_click` и `_select_ability_reward` — выбор игроком и запись способности в runtime-статы игрока.

### Правило base_frame для визуала наград (обязательное)

Во всех reward-flow, где игрок видит карточку `ability`/`artifact`/`cube`, меняется только ассет рамки `base_frame` в зависимости от редкости:

- `обычное` -> `base_frame_normal`;
- `неизвестная`/`null` редкость -> `base_frame_normal` (fallback);
- `необычное` -> `base_frame_copper_`;
- `редкое` -> `base_frame_silver`;
- `уникальное` -> `base_frame_gold`.

Остальные элементы карточки (иконка, описание, выбор, выдача награды, экономика) остаются без изменений.


## 4.3. Runtime-алгоритм награды артефактами (`artifact_+`)

Если после победы на `reward dice` выпадает грань `artifact_+`:

1. генерируются 2 варианта артефактов для выбора игроком;
2. для каждого слота сначала бросается редкость (веса: `обычное 50`, `необычное 25`, `редкое 15`, `уникальное 10`);
3. затем случайно выбирается артефакт нужной редкости;
4. если в этой редкости нет валидных артефактов, выполняется fallback к редкостям ниже (`уникальное -> редкое -> необычное -> обычное`);
5. не-уникальные артефакты могут повторяться (в том числе в рамках одного забега);
6. одинаковый `unique`-артефакт не может быть выдан повторно, если уже есть у игрока или уже выпал в текущем выборе;
7. игрок выбирает один из двух вариантов (`ЛКМ`), после чего артефакт добавляется в `Player.artifacts_runtime` через `Player.grant_artifact`;
8. UI использует `artefact_frame_reward`/`artefact_icon_reward` в `new_battle_table`, где `artefact_icon_reward` динамически получает спрайт выбранного `ArtifactDefinition`;
9. после выбора выполняется переход на глобальную карту.

- `_load_artifact_definitions`;
- `_collect_owned_unique_artifact_ids`;
- `_roll_artifact_reward_rarity`;
- `_pick_artifact_by_rarity_with_fallback`;
- `_render_artifact_reward_cards`, `_apply_artifact_reward_visual` и `_apply_artifact_base_frame_visual`;
- `_resolve_artifact_reward_click` и `_select_artifact_reward`.

## 4.4. Runtime-алгоритм награды кубами (`cube_+`)

Если после победы на `reward dice` выпадает грань `cube_+`:

1. генерируются 2 варианта кубов для выбора игроком;
2. для каждого слота сначала бросается редкость (веса: `обычное 50`, `необычное 25`, `редкое 15`, `уникальное 10`);
3. затем случайно выбирается куб нужной редкости из `res://content/dice/definitions`;
4. если в текущей редкости нет валидных кубов, выполняется fallback по редкостям вниз (`уникальное -> редкое -> необычное -> обычное`);
5. не-уникальные кубы могут повторяться (в том числе в рамках одного забега);
6. одинаковый `unique`-куб не может быть выдан повторно, если уже есть у игрока в любом runtime scope или уже выпал в текущем выборе;
7. игрок выбирает один из двух вариантов (`ЛКМ`), после чего куб добавляется через `Player.grant_runtime_cube` строго в коллекцию своего `scope` (`COMBAT/GLOBAL_MAP/REWARD/MONEY/EVENT`);
8. визуал выбора переиспользует карточку артефакта: вместо спрайта показывается preview реального `Dice`-узла с рамкой `artefact_frame_reward`, поворотом `X=45, Y=10, Z=-120`, смещением `+0.75` по `Y`, масштабом `x3`, без физики;
9. после выбора выполняется переход на глобальную карту.

### Текущие runtime-функции (cube reward)

В `content/combat/reward/post_battle_reward_flow.gd` за `cube_+` flow отвечают:

- `_show_cube_reward_options`;
- `_build_cube_reward_options`;
- `_load_rewardable_cube_definitions`;
- `_collect_owned_unique_cube_ids`;
- `_roll_cube_reward_rarity`;
- `_pick_cube_by_rarity_with_fallback`;
- `_render_cube_reward_cards`, `_apply_cube_reward_visual` и `_apply_cube_base_frame_visual`;
- `_resolve_cube_reward_click` и `_select_cube_reward`.

## 4.5. Runtime-алгоритм денежной награды (`money`)

Если после победы на `reward dice` выпадает грань `money`:

1. выполняется дополнительный физический бросок money-куба(ов) из `player.runtime_money_cubes`;
2. система дожидается остановки дополнительно брошенных кубов;
3. монеты суммируются по верхним граням всех money-кубов, участвовавших в post-battle (стартовый и дополнительный броски);
4. монеты начисляются в `Player.current_coins`;
5. через `2.0` секунды выполняется переход на глобальную карту.

### Текущие runtime-функции (money reward)

В `content/combat/reward/post_battle_reward_flow.gd` за flow `money` отвечают:

- `_grant_money_from_reward_rolls`;
- `_sum_money_from_dice_results`;
- `_count_reward_money_faces`;
- `_roll_bonus_coins_from_runtime_money_dice`;
- `_wait_until_all_dice_stopped`;
- `_return_to_saved_global_map` (с поддержкой `delay_seconds`).

## 5. Базовая конфигурация куба денег

Базовый `money dice` имеет 6 граней со значениями:

- `1`, `2`, `3`, `4`, `5`, `6`.

Значение грани напрямую означает количество выдаваемых монет (`N => +N монет`).

## 6. Границы ответственности

Этот документ описывает только:

- архитектуру двух кубов;
- их место в базовых и runtime-статах;
- момент и правила броска после победы;
- базовые стороны стартовых кубов.

Этот документ намеренно **не описывает**:

- внутреннюю архитектуру артефактов;
- внутреннюю архитектуру карт;
- детальную экономику и хранение валюты;
- генерацию других наград и их таблицы.
