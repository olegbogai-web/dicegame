# Архитектура подсистемы Monster AI

## Статус документа

Этот документ является **специализированным архитектурным стандартом** для подсистемы ИИ монстров.

Он обязателен для любых будущих изменений, связанных с:

- принятием решений монстра в бою;
- фазами начала и завершения хода монстра;
- проверкой доступности способностей по кубам;
- повторным использованием общей логики переноса кубов и активации способностей;
- добавлением новых AI-профилей монстров;
- декомпозицией логики монстров по отдельным файлам;
- интеграцией monster AI с `Combat`, `BattleRoom` и presentation-слоем.

Если `docs/COMBAT_ARCHITECTURE.md` отвечает на вопрос **«как работает общее ядро боя»**, то этот документ отвечает на вопрос **«как именно монстр принимает решение и как это решение передается в общее боевое исполнение без procedural hardcode в `battle_table`»**.

## 1. Цель архитектуры

Цель подсистемы `Monster AI` — сделать поведение монстров:

- **расширяемым** для 200+ монстров;
- **data-driven**, чтобы новые монстры добавлялись в основном через definitions и profile-ресурсы;
- **предсказуемым**, чтобы базовые правила хода были едиными для всех монстров;
- **переиспользуемым**, чтобы перенос кубов и активация способности не переписывались под каждого монстра;
- **тонко интегрированным**, чтобы `battle_table` оставался местом вызова orchestrator-функций, а не местом хранения AI-логики.

Главный принцип:

> **Monster AI описывает решение монстра, а общее боевое ядро и общие action/presentation-сервисы исполняют это решение.**

## 2. Базовые инварианты Monster AI

### 2.1. Монстр не управляет боем напрямую

Monster AI не должен самостоятельно:

- изменять HP цели;
- списывать кубы напрямую из сцены;
- двигать визуальные узлы вручную из monster-specific логики;
- завершать раунд всего боя;
- переключать активного участника вне общего turn-loop.

Monster AI может только:

- запросить оценку доступных действий;
- выбрать следующее намерение;
- сообщить, что нужно завершить ход;
- передать наружу причину решения для логов и отладки.

### 2.2. Общие правила одинаковы для всех монстров

Для всех монстров должны действовать одинаковые системные правила:

- начало хода;
- ожидание остановки кубов;
- проверка, можно ли использовать способность;
- перенос кубов в `dice_place` способности;
- общий визуальный пайплайн использования способности;
- правила завершения хода.

Индивидуальные AI-профили монстров должны менять только:

- приоритеты;
- условия выбора;
- fallback-логику;
- явные сигналы «закончить ход».

### 2.2.1. Краткий реестр приоритетов monster-only способностей

Чтобы при доработках AI-профилей не терялась очередность monster-комбо, фиксируем актуальные приоритеты новых monster-only карт:

- `goblin_inflict_poison` (`content/abilities/definitions/goblin_inflict_poison.tres`) — setup-способность: сначала вешает на себя `poisoner` (1), чтобы усилить следующие наложения `poison`;
- `goblin_poison_strike` (`content/abilities/definitions/goblin_poison_strike.tres`) — follow-up удар: 2 урона + `poison` (2) по врагу, с масштабированием стеков через `poisoner`;
- рекомендуемый порядок в AI: сначала setup (`goblin_inflict_poison`), затем атакующий follow-up (`goblin_poison_strike`), если стоимость по кубам доступна.
- `turtle_durability` (`content/abilities/definitions/turtle_durability.tres`) — defensive setup: накладывает `fortitude` (2) на себя за куб `4/6`;
- `turtle_defense` (`content/abilities/definitions/turtle_defense.tres`) — defensive setup: накладывает `armor` (5) на себя за куб `3/5`;
- `turtle_bite` (`content/abilities/definitions/turtle_bite.tres`) — атакующий fallback: урон `1 + floor(armor(source)/2)` за куб `1/2`;
- рекомендуемый порядок в AI для turtle: сначала `turtle_durability` (пока возможно), затем `turtle_defense` (пока возможно), затем `turtle_bite` (пока возможно).

Для отладки профилей обязательно сохранять reason/debug-сигналы на обе ветки решения (выбор setup, выбор follow-up, невозможность оплатить).

### 2.3. `battle_table` должен оставаться тонким orchestration-слоем

В `battle_table` должны оставаться только вызовы уровня:

- `start_monster_turn(...)`;
- `notify_monster_dice_settled(...)`;
- `run_monster_turn_loop(...)`;
- `request_monster_action(...)`;
- `execute_monster_action(...)`;
- `finish_monster_turn(...)`.

В `battle_table` не должно жить:

- дерево if/else по монстрам;
- правила конкретного тестового монстра;
- внутренняя логика подбора кубов под способность;
- логика анимации под конкретную способность;
- привязка AI к расположению узлов сцены вручную.

## 3. Базовые правила хода монстра

Ниже зафиксированы обязательные системные правила.

## 3.1. Начало хода

В начале хода монстр, **точно так же как игрок**, бросает все свои кубы.

Это означает:

1. создается runtime-состояние хода монстра;
2. в пул хода добавляются все кубы монстра, доступные на этот ход;
3. все эти кубы бросаются через общий боевой/dice pipeline;
4. до остановки кубов AI монстра не начинает decision logic.

## 3.2. Момент старта AI-логики после броска

Монстр **не начинает обрабатывать свою логику, пока двигается хотя бы один куб**.

Строгое правило начала AI-обработки:

> **Monster AI получает право начать обработку действий только после того, как система кубов зафиксировала состояние `all_dice_settled == true` для всех кубов активного монстра.**

Рекомендуемый архитектурный контракт:

- dice/runtime слой публикует событие или callback вида `turn_dice_settled(turn_owner_id)`;
- `MonsterTurnOrchestrator` слушает это событие;
- только после этого orchestrator переводит ход монстра из фазы `waiting_for_roll_settle` в фазу `decision_in_progress`.

До этого момента запрещено:

- вычислять доступные способности;
- выбирать способность;
- переносить кубы;
- запускать use-ability анимацию;
- завершать ход по причине отсутствия подходящих кубов.

### 3.3. Считается, что монстр может использовать способность, если...

Монстр считается способным использовать способность, если выполнены все условия ниже:

1. способность разрешена общими правилами боя;
2. у монстра есть **подходящие кубы** для этой способности;
3. все требуемые кубы уже **полностью остановились**;
4. количество и тип/значение подходящих кубов покрывают всю стоимость способности целиком.

Ключевое правило:

> **Монстр может использовать способность только если у него есть все необходимые кубы для использования способности. Частичная оплата не считается доступностью способности.**

### 3.4. Использование кубов для способности

Монстр может использовать кубы для способностей только по следующим правилам:

- использовать можно только **полностью остановившиеся** кубы;
- кубы должны соответствовать требованиям способности;
- способность считается доступной только при полном покрытии стоимости;
- после выбора способности кубы должны быть выделены из пула, зарезервированы и переданы в общий pipeline исполнения.

### 3.5. Визуальный сценарий использования способности

Если монстр «решает» использовать способность, происходит общий визуальный эффект:

1. выбранные кубы сначала перемещаются на позиции `dice_place` используемой способности;
2. после выставления кубов в слоты добавляется обязательная пауза `0.3s`, и только затем запускается применение эффекта;
3. способность вместе с рамкой и кубами визуально перемещается к цели;
4. кубы исчезают в конце общего эффекта;
5. после завершения визуала общее боевое ядро фиксирует кубы как потраченные и применяет эффект способности.

Обязательное правило:

> **Новый отдельный визуальный hardcode для монстра создавать не нужно; архитектура должна переиспользовать уже реализованные player-facing функции и общий ability-use pipeline.**

Дополнительные тайминговые инварианты для monster turn:

- после каждого применения способности должна быть пауза `0.5s` перед следующим решением AI (чтобы следующий drag кубов не начинался в тот же кадр);
- перед передачей хода игроку должна быть дополнительная пауза `0.3s`;
- формальный старт хода игрока разрешен только после того, как monster-visual pipeline сообщил завершение анимаций (не только после смены turn state).

### 3.6. Конец хода

Монстр передает ход, если выполняется **хотя бы одно** из условий:

1. монстр не может использовать способности;
2. у монстра не осталось кубов;
3. есть четкий сигнал от AI монстра, что ход надо завершить.

Это системное правило должно быть оформлено как общая policy / helper-функция, а не как копия в каждом AI-профиле.

## 4. Целевая фазовая модель хода монстра

Рекомендуемая фазовая модель:

1. `turn_started`
2. `rolling_all_dice`
3. `waiting_for_roll_settle`
4. `decision_in_progress`
5. `executing_action`
6. `post_action_evaluation`
7. `turn_finished`

### 4.1. `turn_started`

Создается `MonsterTurnContext`, привязанный к конкретному монстру и конкретному бою.

### 4.2. `rolling_all_dice`

Общий dice/combat pipeline бросает все кубы монстра.

### 4.3. `waiting_for_roll_settle`

Ход блокируется до тех пор, пока все кубы монстра не перейдут в состояние покоя.

### 4.4. `decision_in_progress`

`MonsterDecisionEngine` получает снимок состояния:

- активный монстр;
- текущий пул кубов;
- список способностей;
- валидные цели;
- уже использованные способности в текущем ходу;
- flags/conditions из статусов и battle state.

На этом этапе AI возвращает одно из двух:

- `use_ability`;
- `end_turn`.

### 4.5. `executing_action`

Если выбрано `use_ability`, управление передается в общий executor способности.

### 4.6. `post_action_evaluation`

После завершения эффекта способность считается исполненной, пул кубов обновляется, затем orchestrator заново проверяет:

- можно ли продолжать ход;
- есть ли еще доступные способности;
- есть ли явный AI-сигнал завершения хода.

Если ход можно продолжать, система возвращается в `decision_in_progress`.

### 4.7. `turn_finished`

Ход завершается с обязательной фиксацией причины:

- `no_usable_abilities`;
- `no_remaining_dice`;
- `ai_requested_end_turn`;
- `combat_finished`.

## 5. Что именно должно быть вынесено в отдельные файлы

Требование пользователя «вынеси логику в отдельные файлы, в `battle_table` должны только вызываться функции» является обязательным архитектурным правилом.

Ниже — рекомендуемая декомпозиция.

## 5.1. Слой orchestration

### `content/combat/monster_ai/monster_turn_orchestrator.gd`

Отвечает за общий loop хода монстра:

- старт хода;
- ожидание остановки кубов;
- запрос решения у AI;
- вызов общего исполнителя способности;
- повторную оценку после действия;
- завершение хода.

### `content/combat/monster_ai/monster_turn_end_policy.gd`

Отвечает только за системное правило конца хода:

- нет доступных способностей;
- нет оставшихся кубов;
- есть AI-сигнал завершения.

## 5.2. Слой decision logic

### `content/combat/monster_ai/monster_decision_engine.gd`

Общий entry point для decision-making.

Принимает `MonsterTurnContext` и делегирует выбор конкретному AI-профилю монстра.

### `content/combat/monster_ai/monster_ai_registry.gd`

Сопоставляет монстра или `monster_ai_id` с нужным AI-профилем/классом.

### `content/combat/monster_ai/profiles/base_monster_ai_profile.gd`

Базовый контракт AI-профиля.

Минимальный интерфейс:

- `decide_next_action(context)`;
- `should_end_turn(context)`;
- опционально `build_debug_reason(context)`.

## 5.3. Shared utility слой

### `content/combat/monster_ai/ability_availability_evaluator.gd`

Проверяет, можно ли использовать способность.

Он должен:

- смотреть только на системные правила доступности;
- не знать о предпочтениях конкретного монстра;
- возвращать пригодные наборы кубов для каждой способности.

### `content/combat/monster_ai/dice_selection_service.gd`

Выбирает конкретный набор кубов, который покроет стоимость способности.

### `content/combat/monster_ai/target_selection_service.gd`

Определяет валидную цель по правилам способности и текущему battle state.

## 5.4. Слой общего исполнения и визуала

### `content/combat/actions/dice_transfer_visual_executor.gd`

Отдельная общая функция/сервис для переноса выбранных кубов в `dice_place` конкретной способности.

Этот сервис должен быть:

- независим от конкретного монстра;
- независим от конкретной способности;
- пригоден для повторного использования игроком и монстрами.

### `content/combat/actions/ability_use_visual_executor.gd`

Отдельная общая функция/сервис для полного визуального сценария использования способности.

Она должна:

- принимать уже выбранную способность, кубы, цель и визуальные ссылки;
- использовать общий визуальный pipeline, уже реализованный для игрока;
- не содержать знания о том, что способность принадлежит тестовому монстру.

### `content/combat/actions/ability_action_executor.gd`

Связывает доменный action resolve и presentation pipeline:

1. переносит кубы в `dice_place`;
2. запускает общий визуал способности;
3. подтверждает трату кубов;
4. вызывает общий resolver эффектов;
5. возвращает результат исполнения в orchestrator.

## 5.5. Presentation binding слой

### `content/combat/presentation/battle_table_action_bridge.gd`

Тонкий адаптер между `battle_table.gd` и общей action/AI-логикой.

Он нужен, чтобы `battle_table.gd` не знал подробностей про:

- выбор AI-профиля;
- подбор кубов;
- детальный порядок вызова визуальных шагов;
- loop повторных действий монстра.

## 6. Контракты данных

## 6.1. `MonsterTurnContext`

Общий контекст хода монстра должен содержать минимум:

- `battle_id`;
- `turn_id`;
- `monster_combatant_id`;
- ссылку на monster definition;
- ссылку на AI profile / `monster_ai_id`;
- текущее состояние кубов монстра;
- список способностей монстра;
- валидные цели;
- историю действий текущего хода;
- флаги фазы хода;
- debug metadata.

## 6.2. `MonsterActionDecision`

Результат решения AI должен быть стандартизирован.

Минимальные поля:

- `decision_type` (`use_ability`, `end_turn`);
- `ability_id`;
- `selected_dice_ids`;
- `target_id`;
- `reason_code`;
- `debug_message`.

## 6.3. `AbilityAvailabilityResult`

Результат системной проверки доступности способности:

- `is_usable`;
- `blocking_reason`;
- `matching_dice_ids`;
- `resolved_target_ids`;
- `requires_all_dice_settled`.

## 7. Правило «можно использовать способность»

Это правило должно быть формализовано централизованно, чтобы все 200+ монстров трактовали его одинаково.

Способность считается usable только если:

1. способность доступна по доменным правилам;
2. найдена хотя бы одна валидная цель;
3. есть полный набор подходящих кубов;
4. каждый выбранный куб находится в состоянии `settled`;
5. ни один из выбранных кубов уже не зарезервирован и не потрачен;
6. после резервирования этих кубов ability executor может выполнить способность без нарушения инвариантов turn state.

Если хотя бы одно условие не выполнено, способность считается недоступной.

---

## 8. Правило завершения хода как общий policy-модуль

Чтобы избежать копирования логики между монстрами, завершение хода должно оцениваться общим policy-слоем.

Псевдоправило:

1. если `ai_profile.should_end_turn(context) == true` → завершить ход;
2. иначе если у монстра нет кубов → завершить ход;
3. иначе если нет ни одной usable способности → завершить ход;
4. иначе ход продолжается.

Важно:

- это **не monster-specific логика**;
- монстр лишь может дать один из входных сигналов;
- окончательное завершение оформляет общий orchestrator/policy.

## 9. Тестовый монстр: обязательный AI-профиль

Для тестового монстра должен существовать отдельный AI-профиль, а не условие внутри `battle_table`.

### 9.1. Поведение тестового монстра

Тестовый монстр должен:

1. использовать способность **«обычный удар»** по игроку, если это возможно;
2. продолжать использовать **«обычный удар»** до тех пор, пока это возможно;
3. закончить свой ход, если использовать **«обычный удар»** больше нельзя.

### 9.2. Как это должно выглядеть архитектурно

Рекомендуемый отдельный файл профиля:

- `content/combat/monster_ai/profiles/test_monster_basic_attack_ai.gd`

Логика профиля должна быть предельно узкой:

- найти у монстра способность `common_attack` / `обычный удар`;
- запросить у общего `AbilityAvailabilityEvaluator`, доступна ли она сейчас;
- если доступна, вернуть `MonsterActionDecision(use_ability)` на игрока;
- если недоступна, вернуть `MonsterActionDecision(end_turn)`.

Профиль не должен:

- двигать кубы;
- запускать анимации;
- сам применять урон;
- сам изменять ход боя.

### 9.3. Почему это важно

Такой подход задает шаблон для масштабирования:

- новый монстр = новый AI-profile или data-конфиг приоритетов;
- общая инфраструктура не меняется;
- тестовый монстр становится референсной минимальной реализацией.

## 10. Стратегия масштабирования на 200+ монстров

Чтобы система была удобной при большом количестве монстров, рекомендуется трехуровневая модель.

## 10.1. Уровень 1 — общие системные сервисы

Одинаковы для всех:

- turn orchestrator;
- end-turn policy;
- ability availability evaluator;
- dice transfer visual executor;
- ability use visual executor;
- action executor.

## 10.2. Уровень 2 — reusable AI building blocks

Переиспользуемые элементы:

- condition evaluators;
- target policies;
- dice selection rules;
- priority rules;
- common reason codes;
- standard action selectors вроде `try_use_first_available_ability_from_list`.

## 10.3. Уровень 3 — monster-specific profiles

Только здесь живут различия монстров:

- порядок проверки способностей;
- пороги HP;
- отношение к защитным действиям;
- желание закончить ход;
- предпочтение определенных целей.

### 10.4. Предпочтительная форма описания новых монстров

Для большинства будущих монстров предпочтительна схема:

- `MonsterDefinition` содержит `monster_ai_id`;
- `monster_ai_id` указывает на AI profile resource или script;
- профиль собирается из data-driven приоритетов и стандартных evaluators;
- код нужен только для действительно уникальных монстров/боссов.

Это позволит избежать 200+ сильно отличающихся вручную написанных классов.

---

## 11. Что не должно находиться в monster-specific AI

В monster-specific AI запрещено помещать:

- lifecycle боевой комнаты;
- порядок хода всех участников;
- низкоуровневую работу со сценой;
- поиск и перемещение `Node3D` вручную;
- прямой вызов резолва урона в обход action executor;
- код визуального эффекта только для одной способности;
- код переноса кубов только для одной способности;
- прямую зависимость на `battle_table.gd` как на место бизнес-логики.

---

## 12. Рекомендуемая схема зависимостей

Правильное направление зависимостей:

`battle_table` → `battle_table_action_bridge` → `monster_turn_orchestrator` → `monster_decision_engine` → `monster_ai_profile`

и отдельно:

`monster_turn_orchestrator` → `ability_action_executor` → `dice_transfer_visual_executor` + `ability_use_visual_executor` + `effect resolver`

Неправильное направление зависимостей:

- `monster_ai_profile` → прямое управление сценой;
- `monster_ai_profile` → прямой вызов VFX;
- `battle_table` → ручной hardcode логики конкретного монстра;
- `ability` → встроенный уникальный монстр-скрипт исполнения.

---

## 13. Минимальный план внедрения без написания логики в `battle_table`

1. Выделить общий orchestrator хода монстра.
2. Выделить общий evaluator доступности способностей.
3. Выделить отдельную общую функцию переноса кубов в `dice_place`.
4. Выделить отдельную общую функцию/use-case визуального использования способности.
5. Подключить их к общему action executor.
6. Создать AI-profile тестового монстра.
7. Оставить в `battle_table` только вызовы orchestration-слоя.

---

## 14. Архитектурный итог

Целевое состояние должно выглядеть так:

- `battle_table` вызывает функции orchestration-слоя и presentation-bridge;
- общие правила хода монстра находятся в `MonsterTurnOrchestrator` и policy-модулях;
- проверка доступности способностей централизована;
- перенос кубов реализован как отдельная общая функция;
- визуал использования способности реализован как отдельная общая функция;
- AI тестового монстра живет в отдельном profile-файле;
- новые монстры добавляются через новые profile/data definitions, а не через рост `battle_table`.

Именно такая архитектура обеспечивает одновременно:

- простоту поддержки;
- красивое разделение файлов;
- удобное масштабирование на большое количество монстров;
- переиспользование уже существующего визуального pipeline игрока;
- минимальный риск превращения monster AI в монолит.
