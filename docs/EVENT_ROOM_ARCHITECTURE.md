# Архитектура EventRoom и событийных сценариев

## Статус документа

Этот документ является **обязательным архитектурным стандартом** для проектирования и реализации событийных комнат (`EventRoom`) и связанных сценариев событий.

Документ уточняет:

- архитектуру нового подкласса комнаты события;
- структуру данных события и вариантов выбора;
- модель исходов для каждого варианта (позитивный / нейтральный / негативный);
- правила броска кубика события и вычисления результата;
- контракт применения последствий к игроку и забегу.

Документ расширяет `docs/ROOM_ARCHITECTURE.md` и не отменяет общие ограничения базовой модели `Room`.

---

## 1. Цель событийной комнаты

`EventRoom` — это специализированная комната, в которой игрок получает текстовое событие, выбирает один из вариантов действия, после чего система бросает специальный кубик события и применяет последствия выбранного варианта.

Событийная комната нужна для:

- нелинейных развилок в забеге;
- риск/награда решений игрока;
- выдачи контента вне боя (артефакты, способности, модификации кубов, запуск цепочек событий и т.д.);
- связки narrative-контента с системными эффектами.

---

## 2. Обязательный состав EventRoom

В каждой комнате события должны присутствовать следующие данные:

1. **Спрайт фона события** — уникальный для конкретного события.
2. **Текст события** — уникальный для конкретного события.
3. **Список вариантов выбора** — набор текстов вариантов (базово 3 варианта, но модель должна поддерживать расширение).
4. **Ссылка на файлы вариантов** — каждый вариант хранится отдельным файлом (resource/definition).

Важно: `EventRoom` хранит ссылки на данные и runtime-состояние, но не содержит hardcode-логики резолва исходов.

---

## 3. Data-driven модель события

Рекомендуемая декомпозиция контента:

- `EventDefinition` — событие целиком (фон, текст, список choice ids, общие условия).
- `EventChoiceDefinition` — один вариант выбора (текст кнопки, конфигурация кубика события, исходы).
- `EventOutcomeDefinition` — один исход варианта (цвет/категория исхода, вес/количество граней, последствия).
- `EventConsequenceDefinition` — атомарные последствия (выдать артефакт, добавить способность, изменить кубы, запустить бой, запустить другое событие и т.д.).

Разделение «один вариант = отдельный файл» является обязательным: каждый `EventChoiceDefinition` должен храниться в своем отдельном `.tres`/definition файле.

---

## 4. Новый подкласс комнаты: EventRoom v2 (целевая роль)

Подкласс `EventRoom` должен выступать как оркестратор жизненного цикла события:

- загрузка `EventDefinition`;
- публикация визуальных данных события в presentation;
- выдача доступных вариантов;
- фиксация выбора игрока;
- запрос броска кубика события;
- резолв исхода;
- применение последствий;
- перевод комнаты в `completed`.

`EventRoom` не должен:

- напрямую вычислять RNG-исход через UI-слой;
- хранить длинные `if/else` по конкретным событиям;
- выполнять побочные эффекты напрямую в scene-скрипте.

---

## 5. Модель варианта события

Каждый вариант события (отдельный файл) содержит:

- `choice_id`;
- `choice_text`;
- параметры доступности (опционально);
- **ровно три категории исходов**:
  - `positive` (зеленый);
  - `neutral` (желтый);
  - `negative` (красный);
- **конфигурацию кубика события для этого варианта**:
  - число зеленых граней;
  - число желтых граней;
  - число красных граней.

Пример конфигурации (иллюстративно):

- Вариант A: 4 зеленых, 2 желтых, 0 красных.
- Вариант B: 1 зеленая, 0 желтых, 5 красных.

Сумма граней должна быть строго положительной.

---

## 6. Кубик события и выбор исхода

После выбора варианта выполняется бросок **Event Dice** по конфигурации выбранного варианта.

### 6.1. Правила

1. Игрок выбирает `EventChoiceDefinition`.
2. Система строит пул граней: `G` зеленых, `Y` желтых, `R` красных.
3. RNG выбирает одну грань из пула равновероятно.
4. Цвет грани определяет исход:
   - зеленый → `positive_outcome`;
   - желтый → `neutral_outcome`;
   - красный → `negative_outcome`.
5. Выбранный исход фиксируется в `EventRoomState` и передается в UI для показа результата.

### 6.2. Инварианты

- Исход выбирается **только** из варианта, выбранного игроком.
- Переход к последствиям выполняется только после фиксации результата.
- Повторный резолв исхода того же выбора в том же прохождении запрещен (idempotency runtime-state).

---

## 7. Последствия исходов

Каждый исход варианта содержит список последствий, которые применяются через отдельный resolver/pipeline.

Поддерживаемые типы последствий (минимум):

- добавление артефакта;
- добавление способности;
- изменение существующих кубов игрока;
- добавление новых кубов;
- запуск боя;
- запуск другого события;
- другие доменные эффекты через расширяемый consequence contract.

### 7.1. Требование к применению

Последствия должны применяться в детерминированном порядке через `EventConsequenceResolver`, а не напрямую из `EventRoom` или UI-кнопки.

Рекомендуемая схема:

1. Сформировать `EventResolutionContext`.
2. Передать в `EventOutcomeResolver`.
3. Получить список `ConsequenceIntent`.
4. Исполнить intents через общий `Run/Combat/Event` orchestration слой.
5. Зафиксировать `EventRoom` как `completed` при успешном завершении.

---

## 8. Presentation-контракт для комнаты события

UI-компонент событийной комнаты обязан получать из домена:

- `background_sprite_ref`;
- `event_text`;
- список `choice_text`;
- состояние доступности кнопок выбора;
- результат броска (цвет + текст результата);
- список кратких примененных последствий (для логирования/информирования игрока).

UI не должен:

- решать исход события самостоятельно;
- изменять состояние игрока напрямую;
- хранить отдельную RNG-логику вне доменного резолвера.

---

## 9. Жизненный цикл EventRoom

Рекомендуемая фаза жизненного цикла:

1. `prepare` — загрузка definitions и проверка валидности данных события.
2. `enter` — показ фона/текста/вариантов.
3. `await_choice` — ожидание выбора игрока.
4. `roll_event_dice` — бросок кубика события.
5. `resolve_outcome` — выбор outcome по цвету.
6. `apply_consequences` — применение системных эффектов.
7. `complete` — завершение комнаты и возврат в flow забега.

---

## 10. Требования к валидации данных

Для каждого `EventChoiceDefinition` валидация должна проверять:

- наличие `choice_id` и текста выбора;
- наличие всех трех категорий исходов (positive/neutral/negative);
- корректность конфигурации граней (неотрицательные числа, сумма > 0);
- валидность ссылок на consequence definitions.

Для `EventDefinition`:

- наличие background sprite ref;
- наличие event text;
- наличие минимум 1 варианта (базово 3 для контент-стандарта);
- валидность ссылок на файлы вариантов.

---

## 11. Расширяемость

Архитектура должна позволять без переработки базового контракта:

- добавлять новые consequence types;
- добавлять preconditions для выбора вариантов (проверка артефактов/ресурсов/состояний);
- подключать разные RNG policy (seeded run RNG, модификаторы удачи, глобальные эффекты);
- добавлять визуальные эффекты на этапе резолва;
- переиспользовать Event Dice модель в других типах комнат.
